*** 1.1 What is Ollama Buddy? [[#What-is-Ollama-Buddy_003f][¶]]
:PROPERTIES:
:CUSTOM_ID: what-is-ollama-buddy
:CLASS: section
:END:
Ollama Buddy is an Emacs package that provides a friendly AI assistant
interface to Ollama, a tool for running large language models (LLMs)
locally on your computer. It allows you to interact with AI models
directly from within Emacs for various tasks such as:

- Code refactoring and explanation
- Writing assistance and proofreading
- Generating Git commit messages
- Dictionary lookups and language assistance
- Custom AI-powered workflows via roles
- Integrating with commercial APIs (OpenAI, Claude, Gemini, Grok,
  Copilot, Codestral, DeepSeek, OpenRouter, OpenCode Go)
- Using Ollama cloud models
- Including files as context in conversations
- Real-time web search to provide current information to LLMs
- Tool calling --- enabling LLMs to read files, run commands, search
  buffers and more

Instead of context-switching to web interfaces or terminal applications,
Ollama Buddy brings the power of local LLMs right into your Emacs
workflow.

<<Communication-Backends>>
*** 2.4 Communication Backends [[#Communication-Backends][¶]]
:PROPERTIES:
:CUSTOM_ID: communication-backends
:CLASS: section
:END:
Ollama Buddy supports two communication backends:

- =network-process (default)= :: Uses Emacs built-in network process.

- =curl= :: Uses external curl command for requests. Useful as a
  fallback when network process has issues.

To switch backends:

#+begin_src example-preformatted
M-x ollama-buddy-switch-communication-backend
#+end_src

or press =C-c e= in the chat buffer. Also available from the transient
menu (C-c O, e) and via the =/backend= slash command.

When the curl backend is active, the header line shows a =⇄= indicator.

To configure curl backend:

#+begin_src example-preformatted
(require 'ollama-buddy-curl)
(setq ollama-buddy-communication-backend 'curl)
(setq ollama-buddy-curl-executable "curl")
(setq ollama-buddy-curl-timeout 300)
#+end_src

<<Provider-Prefixes>>
**** 3.7.1 Provider Prefixes [[#Provider-Prefixes][¶]]
:PROPERTIES:
:CUSTOM_ID: provider-prefixes
:CLASS: subsection
:END:
Prefixes are only used when external providers (OpenAI, Claude, etc.)
are loaded:

- When only Ollama is used :: Models appear without prefix: =llama3:8b=,
  =qwen3-coder:480b-cloud=

- When external providers are loaded :: Prefixes disambiguate providers:

  - =o:= - Ollama local (e.g., =o:llama3:8b=)
  - =u:= - Ollama cloud (e.g., =u:qwen3-coder:480b-cloud=)
  - =a:= - OpenAI (e.g., =a:gpt-4=)
  - =c:= - Claude (e.g., =c:claude-3-opus=)
  - =g:= - Gemini (e.g., =g:gemini-pro=)
  - =k:= - Grok (e.g., =k:grok-1=)
  - =p:= - GitHub Copilot (e.g., =p:gpt-4o=)
  - =s:= - Codestral (e.g., =s:codestral-latest=)

The =☁= cloud symbol provides visual indication for cloud models
regardless of whether prefixes are used.

*** 4.1 Basic Usage [[#Basic-Usage][¶]]
:PROPERTIES:
:CUSTOM_ID: basic-usage
:CLASS: section
:END:
1. Launch Ollama Buddy:

   #+begin_src example-preformatted
   M-x ollama-buddy-role-transient-menu
   #+end_src

   or use your configured keybinding (e.g., =C-c o=).

2. The menu will show available options. Press the corresponding key for
   the action you want.

3. To open the chat interface, press =o= or select "Open Chat".

4. In the chat buffer, type your prompt and press =C-c C-c= to send it.

5. The AI will respond in the chat buffer.

<<Common-Operations>>
*** 4.2 Common Operations [[#Common-Operations][¶]]
:PROPERTIES:
:CUSTOM_ID: common-operations
:CLASS: section
:END:
- Sending text from a file :: Select text in any buffer, then press
  =C-c o= and choose "Send Region" (or press =l=).

- Refactoring code :: Select code, press =C-c o=, then choose "Refactor
  Code" (or press =r=).

- Generating a commit message :: Select your changes, press =C-c o=,
  then choose "Git Commit Message" (or press =g=).

- C-c m :: Change model

- C-c l :: Pull new model Press =C-c m= to switch between available
  models. Press =C-c l= to pull new models from Ollama Hub. Cloud models
  appear in the list marked with =☁=.

- Using the transient menu :: Press =C-c O= for the main transient menu
  with all features organized by category.

- Attaching files :: Press =C-c C-a= to attach a file, or use the
  transient menu “Project” section.

- Project Integration :: Press =C-c P= for the Project sub-menu. Access
  project-specific file attachment and context.

- Peeking at thinking blocks :: While a model is thinking, the
  =*** Thinking= heading is folded. Press =TAB= on the heading to peek
  at accumulated content; press =TAB= again to re-fold.

- Toggling Org/Markdown conversion :: Press =C-c C-o= to switch between
  Org-mode and Markdown output. Also available from the transient menu
  under “Settings”.

- Toggling Global System Prompt :: Press =C-c <= to toggle the global
  system prompt on or off. The header line shows =<= when disabled.

- Getting help :: In the chat buffer, press =C-c h= to display the help
  screen with available commands and models.

*** 5.5 Role Transient Menu [[#Role-Transient-Menu][¶]]
:PROPERTIES:
:CUSTOM_ID: role-transient-menu
:CLASS: section
:END:
The role menu (=ollama-buddy-role-transient-menu=) provides a dynamic
transient popup built from the current role's
=ollama-buddy-command-definitions=. It rebuilds each time it opens, so
switching roles immediately changes the menu.

- Full transient UX with persistent display, help, and discoverability
- Commands grouped into named columns via the =:group= property
- Commands without =:group= appear in a single “Commands” column
- Commands with =:model= show the model name in the description
- Duplicate keys are deduplicated (first definition wins)
- =quit= entries are skipped (transient handles quit natively with
  =C-g=)
- The original minibuffer menu is still available via
  =M-x ollama-buddy-menu=

:PROPERTIES:
:CUSTOM_ID: inline-commands

When typing in the prompt area, pressing =@= triggers a
=completing-read= menu of available inline commands. These insert syntax
templates into the prompt.

Available =@= commands:

- =@search(query)= :: Search the web and attach results.

- =@rag(query)= :: Search a RAG index and attach results.

  completion.

:PROPERTIES:

:PROPERTIES:
:CUSTOM_ID: slash-commands

When typing in the prompt area, pressing =/= triggers a

Available =/= commands (filtered by loaded modules):

- =model= :: Swap the current model (=ollama-buddy--swap-model=).

- =clear= :: Start a new session (=ollama-buddy-sessions-new=).

- =save= :: Save the current session (=ollama-buddy-sessions-save=).

  (=ollama-buddy-toggle-streaming=).

- =reset= :: Reset system prompt to default

- =new= :: Start a new session (=ollama-buddy-sessions-new=).

- =exit= :: Close the chat buffer (=ollama-buddy-exit=).

- =bye= :: Close the chat buffer (=ollama-buddy-exit=), alias for
  =exit=.

- =unload= :: Unload running models from memory
  (=ollama-buddy-unload-model=).

- =set= :: Edit a model parameter (=ollama-buddy-params-edit=).

- =show= :: Show current model info
  (=ollama-buddy-show-raw-model-info=).

- =benchmark= :: Benchmark all models with editable selection
  (=ollama-buddy-benchmark-models=).

- =autoexec= :: Toggle tool auto-execute mode
  (=ollama-buddy-tools-toggle-auto-execute=). Requires
  =ollama-buddy-tools=.

- =unguarded= :: Toggle unguarded mode --- bypass ALL safety prompts for
  tool execution (=ollama-buddy-tools-toggle-unguarded=). Header line
  turns red and shows =☠=. Requires =ollama-buddy-tools=.

- =skill= :: Load a skill as system prompt
  (=ollama-buddy-user-prompts-load=).

- =completion= :: Toggle inline code completions
  (=ollama-buddy-completion-toggle=).

- =manage= :: Open the Model Management buffer
  (=ollama-buddy-manage-models=).

- =project= :: Attach a file from the current project
  (=ollama-buddy-project-attach-file=).

- =init= :: Generate or load a cached project summary
  (=ollama-buddy-project-init=). When no summary exists, one is
  generated and automatically saved to the project root. On subsequent
  uses, the existing summary is attached as context. See
  [[#Project-Integration][Project Integration]].

- =cd= :: Switch the chat buffer's working directory
  (=ollama-buddy-project-switch-directory=). Clears file attachments,
  re-displays the welcome screen with the new project info, and

- =rename= :: Rename the current session

- =logout= :: Sign out from Ollama cloud (=ollama-buddy-cloud-signout=).

  (=ollama-buddy-open-info=).

- =export= :: Open the org-export dispatcher (=org-export-dispatch=) for

  (=ollama-buddy-switch-communication-backend=). Toggles between

  (=ollama-buddy-rewind=). Always presents =completing-read= for prompt

- =plan= :: Start plan mode (=ollama-buddy-plan-start=). Sets a planning

- =plan-next= :: Execute the next =TODO= step in the plan

<<Multishot-Mode>>
*** 7.8 Multishot Mode [[#Multishot-Mode][¶]]
:PROPERTIES:
:CUSTOM_ID: multishot-mode
:CLASS: section
:END:
Multishot mode allows you to send the same prompt to multiple models
simultaneously:

1. Type your prompt in the chat buffer
2. Press =C-c U=
3. Enter the sequence of model letters you want to use (e.g., "a,b,c" to
   use models a, b, and c)
4. Note that each item should be separated with a comma
5. Watch as Ollama Buddy processes your request with each model in
   sequence

Each model in a multishot sequence is subject to a per-model timeout
(default 120 seconds, configurable via
=ollama-buddy-multishot-timeout=). If a model exceeds the timeout, its
request is cancelled, a ‘=TIMED OUT=' marker is inserted, and the
sequence continues with the next model. Set
=ollama-buddy-multishot-timeout= to =nil= to disable.

:PROPERTIES:
:CUSTOM_ID: understanding-context-windows
:CLASS: section

Context windows define how much text (measured in tokens) a model can
process at once. This includes your current prompt, conversation
history, any system prompts, and attached files. Understanding and
managing context is crucial for:

- Preventing errors when context limits are exceeded
- Optimizing model performance for different tasks
- Managing longer conversations efficiently
- Including files without exceeding context limits

:PROPERTIES:
:CUSTOM_ID: context-size-detection
:CLASS: section

Ollama Buddy uses multiple methods to determine a model's context size:

1. Built-in mappings for popular models (llama3, mistral, codellama,
   etc.)
2. Custom context sizes set via the =num_ctx= parameter
3. Manual configuration through interactive commands
4. Fallback to reasonable defaults (4096 tokens) for unknown models

  - Normal font: Under 85% usage
  - Bold and underlined: 85-100% usage
  - Inverted: At or exceeding 100% usage
- Warnings appear before sending prompts that exceed limits

<<How-It-Works>>
*** 10.2 How It Works [[#How-It-Works][¶]]
:PROPERTIES:
:CUSTOM_ID: how-it-works
:CLASS: section
:END:
The web search feature implements a multi-stage pipeline. Content can be
retrieved using two methods, controlled by
=ollama-buddy-web-search-content-source=:

- [[#eww-Mode-_0028Default_002c-Recommended_0029][eww Mode (Default,
  Recommended)]]
- [[#API-Mode-_0028Experimental_0029][API Mode (Experimental)]]

<<eww-Mode-_0028Default_002c-Recommended_0029>>
**** 10.2.1 eww Mode (Default, Recommended) [[#eww-Mode-_0028Default_002c-Recommended_0029][¶]]
:PROPERTIES:
:CUSTOM_ID: eww-mode-default-recommended
:CLASS: subsection
:END:
When =ollama-buddy-web-search-content-source= is set to ='eww= (the
default):

1. *Query to Ollama API*: Search queries are sent to Ollama's Web Search
   API (=https://ollama.com/api/web_search=) via REST with Bearer token
   authentication.
2. *URL Extraction*: The API returns search results containing URLs.
3. *eww/shr Processing*: Each URL is fetched and processed through
   Emacs' built-in eww/shr HTML renderer:
   - HTML is parsed using =libxml-parse-html-region=
   - Main content is extracted (looking for =<article>=, =<main>=, or
     content divs)
   - Content is rendered to clean plain text using =shr-insert-document=
   - Org-mode special characters are escaped (=*= and =#= at line starts
     become =,*= and =,#=)
4. *Context Attachment*: The cleaned text is formatted with org headings
   and attached to the conversation context.
5. *LLM Submission*: The search results are included when sending
   prompts to any configured LLM provider.

This mode produces cleaner, more complete content but requires
additional HTTP requests to fetch each URL.

<<API-Mode-_0028Experimental_0029>>
**** 10.2.2 API Mode (Experimental) [[#API-Mode-_0028Experimental_0029][¶]]
:PROPERTIES:
:CUSTOM_ID: api-mode-experimental
:CLASS: subsection
:END:
When =ollama-buddy-web-search-content-source= is set to ='api=:

1. *Query to Ollama API*: Search queries are sent to the Web Search API.
2. *Direct Content*: Content snippets returned directly by the Ollama
   API are used without fetching individual URLs.
3. *Context Attachment*: The API-provided content is formatted and
   attached to the conversation context.

This mode is faster (no additional HTTP requests) but the content
quality depends on what the Ollama API returns, which may be less
refined than eww-rendered pages. Use this mode when speed is more
important than content completeness.

<<Configuration-2>>
*** 10.3 Configuration [[#Configuration-2][¶]]
:PROPERTIES:
:CUSTOM_ID: configuration
:CLASS: section
:END:
To use web search, you need an API key from Ollama:

1. Visit [[https://ollama.com/settings/keys]]
2. Create a new API key
3. Configure in Emacs:

#+begin_src lisp-preformatted
;; Required: Set your API key
(setq ollama-buddy-web-search-api-key "your-api-key-here")

;; Optional customizations

(setq ollama-buddy-web-search-content-source 'eww)  ; Content source: 'eww (default) or 'api

Note: The web search module is now automatically loaded with Ollama
Buddy. You no longer need to explicitly
=(require 'ollama-buddy-web-search)=.

<<Session-Naming>>
*** 12.7 Session Naming [[#Session-Naming][¶]]
:PROPERTIES:
:CUSTOM_ID: session-naming
:CLASS: section
:END:
When saving a session, Ollama Buddy generates a default name from the
first user message. The name is created by filtering out common English
stop words (“the”, “a”, “is”, etc.) and short words (less than 3
characters), then joining up to 5 remaining key terms with hyphens. A
timestamp prefix is always prepended for uniqueness and chronological
sorting.

For example, a conversation starting with “Tell me about the history of
space exploration” would generate the session name
=2026-03-07-170800-history-space-exploration=.

always prepends a fresh timestamp, so renaming and saving multiple times

<<Rewinding-a-Conversation>>
*** 12.12 Rewinding a Conversation [[#Rewinding-a-Conversation][¶]]
:PROPERTIES:
:CUSTOM_ID: rewinding-a-conversation
:CLASS: section
:END:
You can rewind a conversation to a previous prompt in several ways:

- =C-u C-u C-c C-c= --- when point is on a prompt heading, rewinds
  directly to that prompt. Otherwise falls back to =completing-read=.
- =/rewind= --- slash command that always presents =completing-read=.
- =C-c O= ⇒ =u= --- transient menu “Rewind”, always presents
  =completing-read=.

When using =C-u C-u C-c C-c= on a heading, you can navigate there with:

- Fold the buffer to show only prompt headings (=S-TAB= to cycle global
  visibility).
- Use =imenu= to jump to a specific prompt.
- Simply scroll up and place point on a ‘=>> PROMPT:=' heading.

After confirmation, everything from that prompt onward is removed and
the conversation history is truncated to match. The prompt text is
pre-filled in a fresh prompt area so you can edit and resend it.

<<Role-File-Naming-Convention>>
*** 14.2 Role File Naming Convention [[#Role-File-Naming-Convention][¶]]
:PROPERTIES:
:CUSTOM_ID: role-file-naming-convention
:CLASS: section
:END:
The file naming convention is critical to understand how roles, preset
files, and menu configurations work together:

- Required filename format :: =ollama-buddy--preset__ROLE-NAME.el=

  - The double underscore =__= separates the prefix from your role name
  - The role name portion becomes the identifier shown when switching
    roles
  - Example: =ollama-buddy--preset__programmer.el= creates a role named
    "programmer"

This naming convention is how Ollama Buddy discovers and identifies role
files. When you run =ollama-buddy-roles-switch-role=, the system:

1. Scans the =ollama-buddy-roles-directory= for files matching the
   pattern
2. Extracts the role name from each filename (the part after =__=)
3. Presents these names in the role selection interface
4. When selected, loads the corresponding file which redefines
   =ollama-buddy-command-definitions=
5. This redefinition immediately changes the available commands in your
   Ollama Buddy menu

The relationship chain works like this:

#+begin_src example-preformatted
ollama-buddy--preset__ROLE-NAME.el → Defines ollama-buddy-command-definitions → Controls menu content
#+end_src

When creating roles using the interactive role creator (=C-c E=), this
naming convention is automatically handled for you. When creating roles
manually, you must follow this pattern for Ollama Buddy to recognize
your role files correctly.

<<Interactive-Role-Creator>>
**** 14.4.1 Interactive Role Creator [[#Interactive-Role-Creator][¶]]
:PROPERTIES:
:CUSTOM_ID: interactive-role-creator
:CLASS: subsection
:END:
The most user-friendly approach:

1. Press =C-c E= or run =M-x ollama-buddy-role-creator-create-new-role=
2. Enter a name for your role (e.g., "programmer")
3. Enter the number of menu columns (default: 2)

   - Specify a command name (e.g., "refactor-code")

<<Manual-Role-Creation>>
**** 14.4.2 Manual Role Creation [[#Manual-Role-Creation][¶]]
:PROPERTIES:
:CUSTOM_ID: manual-role-creation
:CLASS: subsection
:END:
For more advanced customization, create role files manually:

1. Create a file named =ollama-buddy--preset__your-role-name.el= in your
   =ollama-buddy-roles-directory=
2. Structure your file like this:

#+begin_src example-preformatted
;; ollama-buddy preset for role: programmer
(require 'ollama-buddy)

(setq ollama-buddy-command-definitions
  '(
    ;; Standard commands
    (open-chat
     :key ?o
     :description "Open chat buffer"
     :group "General"
     :action ollama-buddy--open-chat)

    (switch-role
     :key ?R
     :description "Switch roles"
     :group "General"
     :action ollama-buddy-roles-switch-role)

    ;; Custom commands for this role
    (refactor-code
     :key ?r
     :description "Refactor code"
     :group "Code"
     :model "codellama:7b"
     :prompt "Refactor this code to improve readability and efficiency:"
     :system "You are an expert software engineer who improves code quality."
     :action (lambda () (ollama-buddy--send-with-command 'refactor-code))
     :destination in-buffer)

    (explain-code
     :key ?e
     :description "Explain code"
     :group "Code"
     :model "deepseek-r1:7b"
     :prompt "Explain what this code does in detail:"
     :system "You are a programming teacher who explains code clearly."
     :action (lambda () (ollama-buddy--send-with-command 'explain-code))
     :destination chat)

    (git-commit
     :key ?g
     :description "Git commit message"
     :group "Utility"
     :prompt "Write a concise git commit message for these changes:"
     :system "You are a version control expert who creates clear commit messages."
     :action (lambda () (ollama-buddy--send-with-command 'git-commit))
     :destination chat)

    ;; System commands
    (custom-prompt
     :key ?e
     :description "Custom prompt"
     :group "System"
     :action ollama-buddy--menu-custom-prompt)
    ))
#+end_src

The =:group= property is optional. Commands sharing the same =:group=
value appear together in a named column in the transient role menu
(=ollama-buddy-role-transient-menu=). Presets without any =:group=
properties still work --- all commands appear in a single “Commands”
column for full backwards compatibility.

<<Per_002dCommand-Response-Destination>>
*** 14.6 Per-Command Response Destination [[#Per_002dCommand-Response-Destination][¶]]
:PROPERTIES:
:CUSTOM_ID: per-command-response-destination
:CLASS: section
:END:
The optional =:destination= property on a command definition overrides
the global =ollama-buddy-in-buffer-replace= toggle for that specific
command:

- =in-buffer= :: Always stream the response back into the source buffer,
  replacing the selected region (requires =ollama-buddy-rewrite= to be
  loaded). The global toggle is ignored.

- =chat= :: Always open the chat buffer, regardless of the global
  toggle.

- =nil (or absent)= :: Honour =ollama-buddy-in-buffer-replace= as before
  --- this is the default and preserves full backwards compatibility.

Use =:destination in-buffer= for commands that /transform/ selected text
(refactoring, proofreading, translation, style rewrites) and
=:destination chat= for commands that /analyse or generate new content/
about the selection (code explanations, commit messages, dictionary
lookups).

#+begin_src example-preformatted
;; Transforms selected text --- always rewrites in-buffer
(refactor-code
 :key ?r
 :description "Refactor code"
 :prompt "Refactor this code:"
 :action (lambda () (ollama-buddy--send-with-command 'refactor-code))
 :destination in-buffer)

;; Generates new content --- always goes to chat
(explain-code
 :key ?e
 :description "Explain code"
 :prompt "Explain this code:"
 :action (lambda () (ollama-buddy--send-with-command 'explain-code))
 :destination chat)

;; No :destination --- honours the global toggle
(describe-code
 :key ?c
 :description "Describe code"
 :prompt "Describe this code:"
 :action (lambda () (ollama-buddy--send-with-command 'describe-code)))
#+end_src

All built-in presets ship with =:destination= annotations. Commands
added via =ollama-buddy-update-menu-entry= or =add-to-list= on
=ollama-buddy-command-definitions= work identically --- simply include
or omit the =:destination= key as appropriate.

:PROPERTIES:

Ollama Buddy integrates with multiple commercial AI services, each
loaded on-demand:

- OpenAI (prefix: =a:=) :: Load with

- Claude/Anthropic (prefix: =c:=) :: Load with
  =(require 'ollama-buddy-claude nil t)=

- Google Gemini (prefix: =g:=) :: Load with
  =(require 'ollama-buddy-gemini nil t)=

- Grok/X (prefix: =k:=) :: Load with
  =(require 'ollama-buddy-grok nil t)=

- GitHub Copilot (prefix: =p:=) :: Load with
  =(require 'ollama-buddy-copilot nil t)=

- Codestral/Mistral (prefix: =s:=) :: Load with
  =(require 'ollama-buddy-codestral nil t)=

- DeepSeek (prefix: =d:=) :: Load with
  =(require 'ollama-buddy-deepseek nil t)=

- OpenRouter (prefix: =r:=) :: Load with
  =(require 'ollama-buddy-openrouter nil t)=

- OpenCode Go (prefix: =n:=) :: Load with
  =(require 'ollama-buddy-opencode nil t)=. All 14 subscription models
  register under one prefix with one shared API key; per-model dispatch
  routes OpenAI-shape models (GLM, Kimi, MiMo, Qwen) through
  =/chat/completions= and Anthropic-shape models (DeepSeek V4, MiniMax)
  through =/messages=. Note: this is the flat-rate Go subscription,
  distinct from OpenCode's pay-as-you-go Zen API.

- Generic OpenAI-compatible server (prefix: =l:=, configurable) :: Load
  with =(require 'ollama-buddy-openai-compat nil t)=. Works with LM
  Studio, llama.cpp, vLLM, Jan, and any
  =/v1/chat/completions=-compatible endpoint. No API key needed for
  local servers.

Local Ollama models use the =o:= prefix when remote models are
available, or no prefix when only local models are present.

<<Status-Line-Indicators>>
*** 15.2 Status Line Indicators [[#Status-Line-Indicators][¶]]
:PROPERTIES:
:CUSTOM_ID: status-line-indicators
:CLASS: section
:END:
The header line shows compact status indicators:

- =✈= - Airplane mode active (local Ollama only, internet blocked)
- =☁= - Currently using an Ollama cloud model
- =⚒= - Tool calling enabled AND model supports tools
- =⚡= - Tool auto-execute enabled (unsafe tools still prompt)
- =☠= - Unguarded mode: ALL safety prompts bypassed (header line turns
  red)
- =⊙= - Current model supports vision
- =✦= - Thinking enabled and model supports reasoning
- =✧= - Thinking disabled (model supports reasoning but =think= is off)
- =≡N= - Number of file attachments (e.g. =≡3= means 3 files)
- =♁N= - Number of web search results attached
- =↓= - Auto-scroll enabled (buffer follows streaming output)
- =x= - Streaming disabled
- =<= - Global system prompt disabled
- =H5/10= - History count (when enabled via
  =ollama-buddy-show-history-indicator=)
- =V= - Reasoning/thinking sections hidden
- Provider letters: =a= (OpenAI), =c= (Claude), =g= (Gemini), =k=
  (Grok), =p= (Copilot), =s= (Codestral), =d= (DeepSeek), =r=
  (OpenRouter), =n= (OpenCode Go), =l= (OpenAI-compat)

Example: =☁⚒⊙✦≡2♁1acgkpsdrn= means cloud model with tools enabled,
vision support, thinking support, 2 file attachments, 1 web search, and
all nine external providers loaded.

changed. Re-copy the fresh value from your browser and update
=ollama-buddy-cloud-session-token=.

<<Overview-5>>
*** 17.1 Overview [[#Overview-5][¶]]
:PROPERTIES:
:CUSTOM_ID: overview-5
:CLASS: section
:END:
The =/launch= slash command lets you launch any Ollama model (local or
cloud) in an external terminal with a frontend tool such as Claude Code.

There are two types of agents:

- *Standard agents* :: Use =ollama launch FRONTEND --model MODEL=. These
  are built-in Ollama integrations (Claude Code, Codex, Pi, etc.).

- *Direct agents* :: Invoked as standalone executables in a terminal,
  bypassing =ollama launch= entirely. This allows external tools that
  are not built-in Ollama integrations to participate in the launch
  workflow. Direct agents only require their own executable on PATH ---
  they do not depend on =ollama= being installed.

<<Usage-1>>
*** 17.2 Usage [[#Usage-1][¶]]
:PROPERTIES:
:CUSTOM_ID: usage-1
:CLASS: section
:END:
- Slash command :: Type =/launch= in the chat prompt area. You will be
  prompted to select a model, then a frontend (if more than one is
  configured).

- Model Management :: Each model in the =C-c M= Model Management buffer
  has a =Launch= button (when frontends are detected on PATH).

- Interactive (from chat) :: =M-x ollama-buddy-launch=

- Standalone (from anywhere) :: =M-x ollama-buddy-launch-external= or
  press v in the main transient menu (C-c O). This does not require the
  Ollama Buddy chat buffer --- it launches an agent directly from the
  current directory. In Dired, the terminal opens in the displayed
  directory. The model prompt shows the target directory for
  confirmation.

All launch methods produce fully detached terminal sessions that survive
closing Emacs.

<<Terminal-Auto_002dDetection>>

:PROPERTIES:
:CUSTOM_ID: terminal-auto-detection
:CLASS: section
:END:
When =ollama-buddy-launch-terminal= is =nil= (the default), the terminal
emulator is auto-detected in this order:

1. =TERMINAL= environment variable
2. =TERM_PROGRAM= environment variable (set by modern terminals)
3. =xdg-terminal-exec= (freedesktop.org standard)
4. PATH scan of common terminals: kitty, alacritty, foot, wezterm,
   ghostty, gnome-terminal, kgx, konsole, xfce4-terminal, mate-terminal,
   lxterminal, tilix, terminator, st, urxvt, xterm

Each terminal is paired with its correct command-line flag (=-e= for
most, =--= for gnome-terminal, =start --= for wezterm).

<<Small-Model-Warning>>
*** 17.6 Small Model Warning [[#Small-Model-Warning][¶]]
:PROPERTIES:
:CUSTOM_ID: small-model-warning
:CLASS: section
:END:
Coding agents such as Claude Code, OpenCode, Codex, and others prepend
*large system prompts* (20,000+ tokens) to every request. These prompts
contain tool definitions, safety guardrails, formatting instructions,
and other context that the agent needs to function.

This means that even a simple message like “hello” actually requires the
model to process over 20,000 tokens before it can generate a single word
of output. On small models (below /\sim/7B parameters), especially
without a dedicated GPU, this can cause:

- The model appearing *completely frozen* for minutes at a time
- *No visible output* while the model slowly encodes the system prompt
- *Extremely slow responses* once generation finally begins
- *Token counts showing zero* in the agent UI while prompt encoding is
  still in progress

**** Recommendations [[#Recommendations][¶]]
:PROPERTIES:
:CUSTOM_ID: Recommendations
:CLASS: subheading
:END:
- Use models with *7B+ parameters* for agent workflows (14B+ preferred)
- A *dedicated GPU with sufficient VRAM* is strongly recommended --- CPU
  inference with large agent contexts is impractical for interactive use
- Consider *Ollama cloud models* (e.g. =qwen3-coder:480b-cloud=) which
  offload compute to Ollama's infrastructure
- If a model appears frozen after launch, run =ollama ps= in a terminal
  to see whether the model is loaded and how much is offloaded to CPU

When you launch a model below the threshold set in
=ollama-buddy-launch-small-model-threshold= (default 7B), Ollama Buddy
will display a confirmation warning explaining the potential performance
impact. This warning is skipped for cloud models, which run on remote
infrastructure.

<<Troubleshooting-2>>
*** 17.8 Troubleshooting [[#Troubleshooting-2][¶]]
:PROPERTIES:
:CUSTOM_ID: troubleshooting-1
:CLASS: section
:END:
**** npm permission errors (Pi and other npm-based agents) [[#npm-permission-errors-_0028Pi-and-other-npm_002dbased-agents_0029][¶]]
:PROPERTIES:
:CUSTOM_ID: npm-permission-errors-_0028Pi-and-other-npm_002dbased-agents_0029
:CLASS: subheading
:END:
Some agents (e.g. Pi) are installed via =npm install -g=. On Linux, the
default global prefix is =/usr/lib/node_modules/=, which requires root
access. If =ollama launch pi= fails with =EACCES: permission denied=,
configure npm to use a user-local directory:

#+begin_src example-preformatted
mkdir -p ~/.npm-global/bin
npm config set prefix ~/.npm-global
#+end_src

Then add =~/.npm-global/bin= to your =PATH=.

For *fish shell*:

#+begin_src example-preformatted
fish_add_path ~/.npm-global/bin
echo 'fish_add_path ~/.npm-global/bin' >> ~/.config/fish/config.fish
#+end_src

For *bash/zsh*, add to =~/.bashrc= or =~/.zshrc=:

#+begin_src example-preformatted
export PATH="$HOME/.npm-global/bin:$PATH"
#+end_src

After this, re-run =ollama launch pi= and the web search plugin will
install correctly.

<<Overview-6>>
*** 18.1 Overview [[#Overview-6][¶]]
:PROPERTIES:
:CUSTOM_ID: overview-6
:CLASS: section
:END:
Ollama Buddy can drive structured LLM output to generate
=simply-annotate= annotation databases across whole directories of
source code. The batch workflow attaches one file at a time, forces the
model into a JSON schema via the “Annotate Project” skill, and applies
the returned JSON to a =.simply-annotations.el= database at the project
root.

<<Commands>>
*** 18.2 Commands [[#Commands][¶]]
:PROPERTIES:
:CUSTOM_ID: commands
:CLASS: section
:END:
- =M-x ollama-buddy-annotate-directory= :: Or =/annotate-dir=. Prompts
  for a directory, queues every matching source file (see
  =ollama-buddy-annotate-directory-extensions=), and sends one
  annotation request per file. A post-response hook auto-applies the
  returned JSON and dequeues the next file. With a prefix argument,
  descends recursively.

- =M-x ollama-buddy-annotate-directory-cancel= :: Or =/annotate-cancel=.
  Stops an in-flight batch, removes the post-response hook, and clears
  the queue.

- =M-x ollama-buddy-annotate-apply-last-response= :: Or C-c A in the
  chat buffer, or =/annotate=. Applies the JSON from the last response
  in the chat buffer to the annotation database for a file you pick.
  Used both manually and internally by the batch loop.

<<Context_002daware-batching>>
*** 18.3 Context-aware batching [[#Context_002daware-batching][¶]]
:PROPERTIES:
:CUSTOM_ID: context-aware-batching
:CLASS: section
:END:
Each queued file's estimated token cost (characters / 4) is checked
against the current model's context window. Files whose estimate exceeds
the budget are skipped with a message. The reserve for the response is
configurable:

- =ollama-buddy-annotate-context-reserve= :: Fraction of the context
  window reserved for the response and overhead (default 0.5). A file is
  allowed when =estimated-tokens <= context-size * (1 - reserve)=.

<<Structured-output-schema>>
*** 18.4 Structured output schema [[#Structured-output-schema][¶]]
:PROPERTIES:
:CUSTOM_ID: structured-output-schema
:CLASS: section
:END:
The batch loads the “Annotate Project” user-prompt skill (configurable
via =ollama-buddy-annotate-directory-skill=), which sets the active
system prompt and installs a JSON schema via its =#+FORMAT:= header. The
schema requires one annotations array whose items have =sl=, =el=,
=text=, and =tag= (one of =file=, =defun=, or =line=). Structured output
guarantees the model returns exactly this shape, which makes unattended
batching reliable.

You can inspect the active schema at any time with C-c C-s
(=ollama-buddy-show-system-prompt-info=) -- the system-prompt buffer now
shows the loaded =#+FORMAT:= schema alongside the prompt content when
structured output is armed.

<<Database-shape>>
*** 18.5 Database shape [[#Database-shape][¶]]
:PROPERTIES:
:CUSTOM_ID: database-shape
:CLASS: section
:END:
Annotations are written as full =simply-annotate= thread structures
(=id=, =created=, =status=, =priority=, =tags=, =comments=) rather than
plain strings, so the =tag= from the schema lands in the thread's =tags=
list directly. When the database strategy is =project= (default), file
keys are stored relative to the project root to keep the database
portable between clones.

- =ollama-buddy-annotate-db-strategy= :: Where to write the database.
  =project= (default) writes to =.simply-annotations.el= at the project
  root; =global= writes to =simply-annotations.el= in
  =user-emacs-directory= and stores absolute paths.

- =ollama-buddy-annotate-directory-extensions= :: List of file-name
  extensions considered by =ollama-buddy-annotate-directory=. Covers
  common compiled and scripting languages; extend it for
  project-specific file types.

<<Overview-9>>
*** 21.1 Overview [[#Overview-9][¶]]
:PROPERTIES:
:CUSTOM_ID: overview-9
:CLASS: section
:END:
Reasoning models (such as DeepSeek-R1, QwQ, and others) produce an
internal chain-of-thought before giving their final answer. Ollama Buddy
renders this reasoning content as a native org-mode heading that folds
automatically, so the thinking process is available for inspection but
does not clutter the conversation.

<<How-It-Works-3>>
*** 21.2 How It Works [[#How-It-Works-3][¶]]
:PROPERTIES:
:CUSTOM_ID: how-it-works-3
:CLASS: section
:END:
When a model emits =<think>= tags or uses the DeepSeek
=message.thinking= API field, Ollama Buddy:

1. Inserts a =*** ✦ Think= org heading in the chat buffer.
2. Streams the reasoning content under that heading in real time.
3. When reasoning ends, folds the heading with =outline-hide-subtree=
   and inserts a =*** ✦ Response= heading to mark the start of the
   actual answer.

The =✦= symbol appears in the header line and prompt line when thinking
is enabled for the active model. When thinking is disabled (via
=/think=), the symbol changes to =✧=.

<<Key-Bindings>>
*** 21.3 Key Bindings [[#Key-Bindings][¶]]
:PROPERTIES:
:CUSTOM_ID: key-bindings
:CLASS: section
:END:
- TAB :: On a =*** Think= heading, unfold to peek at thinking content;
  press again to re-fold (standard org-mode behavior). Works during
  streaming and after completion.

<<Configuration-7>>
*** 21.4 Configuration [[#Configuration-7][¶]]
:PROPERTIES:
:CUSTOM_ID: configuration-5
:CLASS: section
:END:
- =ollama-buddy-thinking-enabled= :: When non-nil (default =t=), the
  Ollama API =think= parameter is sent to reasoning-capable models so
  they produce a thinking block. When nil, models skip reasoning
  entirely for faster, cheaper responses. Toggle with =/think=,
  transient T, or =M-x ollama-buddy-toggle-thinking=. The header line
  shows =✦= when enabled or =✧= when disabled.

- =ollama-buddy-collapse-thinking= :: When non-nil (default =t=),
  thinking content is automatically collapsed to the =*** ✦ Think=
  heading after the reasoning block ends.

- =ollama-buddy-hide-reasoning= :: When non-nil and
  =ollama-buddy-collapse-thinking= is nil, the thinking block is hidden
  entirely (replaced by “Thinking...”) rather than kept in the buffer.

- =ollama-buddy-thinking-models= :: Static list of model name substrings
  that are known reasoning models. Auto-detection from the Ollama
  =/api/show= capabilities array is also performed.

<<Overview-10>>
*** 23.1 Overview [[#Overview-10][¶]]
:PROPERTIES:
:CUSTOM_ID: overview-10
:CLASS: section
:END:
In-buffer replace streams LLM output directly into the source buffer
rather than the chat buffer. When active, commands that operate on a
selected region (Refactor, Proofread, Git Commit, Dictionary Lookup,
Synonym, Describe Code, or any custom role command) replace the
selection in-place with the model's response, shown with a green
highlight pending your decision.

The existing chat buffer workflow is completely unchanged when the
toggle is off (the default). All existing commands, prompts, system
prompts, and model selection logic are unaffected --- only the output
destination changes.

*Per-command override:* individual commands can pin their destination
regardless of the global toggle by setting the =:destination= property
in =ollama-buddy-command-definitions=. A command with
=:destination in-buffer= always rewrites in-buffer; one with
=:destination chat= always opens the chat buffer. Commands without a
=:destination= property continue to honour the global toggle. See
[[#Roles-and-Commands][Per-Command Response Destination]], for details.

<<Enabling-In_002dBuffer-Replace>>
*** 23.2 Enabling In-Buffer Replace [[#Enabling-In_002dBuffer-Replace][¶]]
:PROPERTIES:
:CUSTOM_ID: enabling-in-buffer-replace
:CLASS: section
:END:
Toggle via the transient Actions menu (C-c O, key W) or:

#+begin_src example-preformatted
M-x ollama-buddy-toggle-in-buffer-replace
#+end_src

The =✎= symbol appears in the header line and prompt line while the mode
is active. The custom role menu footer shows a live
‘=In-Buffer Replace [ON] / [OFF]=' toggle.

<<Basic-Workflow>>
*** 23.3 Basic Workflow [[#Basic-Workflow][¶]]
:PROPERTIES:
:CUSTOM_ID: basic-workflow
:CLASS: section
:END:
1. Toggle in-buffer replace on (C-c O ⇒ W).
2. In any source buffer, select a region of text (e.g. a function body,
   a paragraph).
3. Invoke a role command (e.g. Refactor from the Developer role).
4. A dimmed ‘=[Rewriting...]=' placeholder replaces the selection while
   connecting.
5. The model's response streams in with a green highlight.
6. When streaming finishes, the mode line shows ‘= [Rewrite?]='.
7. Press C-c C-c to *accept* (keep new text, remove highlight).
8. Press C-c C-k to *reject* (discard new text and restore the
   original).

Cancelling mid-stream with C-c C-k stops the network process and
restores the original selection immediately.

<<Inline-Diff-View>>
*** 23.4 Inline Diff View [[#Inline-Diff-View][¶]]
:PROPERTIES:
:CUSTOM_ID: inline-diff-view
:CLASS: section
:END:
After streaming completes, press C-c d to open an inline diff view
inside the same buffer:

#+begin_src example-preformatted
[new text — green background]
  [changed words — brighter green overlay]

── Original ────────────────────────────
[original text — yellow background]
  [removed words — red strikethrough overlay]
────────────────────────────────────────
#+end_src

Word-level differences are highlighted by =smerge-refine-regions=: green
overlays mark added or changed words in the new text; red strikethrough
overlays mark words present in the original but absent from the rewrite.

Press C-c d again to hide the diff block without accepting or rejecting.
Accepting (C-c C-c) or rejecting (C-c C-k) automatically removes the
diff block.

<<Key-Bindings-_0028while-rewrite-is-pending_0029>>
*** 23.5 Key Bindings (while rewrite is pending) [[#Key-Bindings-_0028while-rewrite-is-pending_0029][¶]]
:PROPERTIES:
:CUSTOM_ID: key-bindings-while-rewrite-is-pending
:CLASS: section
:END:
- C-c C-c :: Accept the rewrite --- keep the streamed text and remove
  the highlight.

- C-c C-k :: Reject the rewrite --- stop streaming (if still in
  progress) and restore the original selected text.

- C-c d :: Toggle the inline diff view. Shows the original text below
  the new text with word-level smerge highlighting. Press again to hide.

<<Automatic-Output-Polishing>>
*** 23.6 Automatic Output Polishing [[#Automatic-Output-Polishing][¶]]
:PROPERTIES:
:CUSTOM_ID: automatic-output-polishing
:CLASS: section
:END:
Two automatic adjustments help keep in-buffer results clean:

- *Clean-output tone* --- when in-buffer replace is active a “Return
  only the requested content” instruction is automatically prepended to
  the effective system prompt, suppressing introductory phrases and
  commentary.
- *Code-fence stripping* --- if the model wraps its response in markdown
  code fences (‘=```lang … ```='), the fences are stripped before
  insertion, leaving only the inner content.

<<Implementation-Notes>>
*** 23.7 Implementation Notes [[#Implementation-Notes][¶]]
:PROPERTIES:
:CUSTOM_ID: implementation-notes
:CLASS: section
:END:
The feature is implemented in =ollama-buddy-rewrite.el=, loaded
optionally:

#+begin_src example-preformatted
(require 'ollama-buddy-rewrite nil t)
#+end_src

The module creates a direct =make-network-process= connection to Ollama
(=/api/chat=) and streams JSON responses using the same line-delimited
format as the main chat buffer filter. Two Emacs markers track the
rewrite region:

- A /non-advancing/ start marker (type =nil=) --- stays before inserted
  text.
- An /advancing/ end marker (type =t= during streaming, switched to
  =nil= on completion) --- moves past each newly inserted chunk, then
  fixed so the diff block insertion does not displace it.

The inline diff block is inserted as real buffer text at the fixed end
marker position. =smerge-refine-regions= is called with both the
original block and the new text residing in the same buffer, which is
the standard within-buffer usage of that function. Original text is
preserved in the rewrite state plist for rejection; no undo history
manipulation is required.

<<Limitations>>
*** 23.8 Limitations [[#Limitations][¶]]
:PROPERTIES:
:CUSTOM_ID: limitations
:CLASS: section
:END:
- Only works with local Ollama models (not remote providers).
- Only one rewrite can be pending at a time.
- The streaming response is inserted verbatim; no markdown-to-org
  conversion is performed (unlike the chat buffer) --- the fence
  stripper handles the common code-block case.

Create a file at /tmp/hello.txt with Hello World.

Evaluate (length (buffer-list)) and tell me how many buffers I have.
#+end_src

<<How-It-Works-4>>
*** 24.5 How It Works [[#How-It-Works-4][¶]]
:PROPERTIES:
:CUSTOM_ID: how-it-works-4
:CLASS: section
:END:
When tool calling is enabled and a tool-capable model is selected, the
conversation flow works as follows:

1. You send a prompt (e.g., “What files are in my home directory?”).
2. Ollama Buddy sends the prompt along with the tool schemas to the
   model.
3. The model decides whether to call a tool. If so, it returns a tool
   call request instead of text.
4. Ollama Buddy executes the requested tool (e.g., =list_directory= with
   path =~=).
5. The tool result is sent back to the model as a =tool= role message.
6. The model generates a natural language response incorporating the
   tool output.
7. Steps 3--6 may repeat if the model needs multiple tools, up to
   =ollama-buddy-tools-max-iterations=.

This loop is transparent to the user --- you see the final response in
the chat buffer.

- [[#Model-Compatibility][Model Compatibility]]

<<Model-Compatibility>>
**** 24.5.1 Model Compatibility [[#Model-Compatibility][¶]]
:PROPERTIES:
:CUSTOM_ID: model-compatibility
:CLASS: subsection
:END:
Tools are only sent to models that support them. If you have tools
enabled but switch to a model that doesn't support tool calling:

- The request proceeds normally without tools (no error)
- The header line =⚒= indicator disappears (tools enabled but not
  applicable)
- The prompt line =⚒= indicator also disappears (model doesn't support
  tools)
- When you switch back to a tool-capable model, tools become active
  again

This allows you to leave tools enabled globally while freely switching
between models.

<<Registering-Custom-Tools>>
*** 24.7 Registering Custom Tools [[#Registering-Custom-Tools][¶]]
:PROPERTIES:
:CUSTOM_ID: registering-custom-tools
:CLASS: section
:END:
Use =ollama-buddy-tools-register= to add your own tools:

#+begin_src example-preformatted
(ollama-buddy-tools-register
 'my-tool                          ; NAME (symbol or string)
 "Description of what the tool does" ; DESCRIPTION
 '((type . "object")               ; PARAMETERS (JSON schema)
   (required . ["param1"])
   (properties . ((param1 . ((type . "string")
                              (description . "Parameter description"))))))
 (lambda (args)                    ; FUNCTION (receives alist, returns string)
   (let ((param1 (alist-get 'param1 args)))
     (format "Result: %s" param1)))
 t)  ; SAFE flag: t = safe (read-only), nil = unsafe
#+end_src

To remove a tool:

#+begin_src example-preformatted
(ollama-buddy-tools-unregister 'my-tool)
#+end_src

The tool function receives an alist of arguments parsed from the model's
JSON response and must return a string result. If the function returns a
non-string value, it is formatted with =%S=.

<<Tool-Indicators>>
*** 24.9 Tool Indicators [[#Tool-Indicators][¶]]
:PROPERTIES:
:CUSTOM_ID: tool-indicators
:CLASS: section
:END:
The =⚒= (hammer and wrench) indicator appears in two places with
different meanings:

- Header line :: Shows =⚒= when tool calling is currently /enabled/
  (toggled on with C-c SPC). This indicates that tool calls will be sent
  to the model.

- Prompt line :: Shows =⚒= when the current model /supports/ tool
  calling. This appears regardless of whether tools are enabled, helping
  you identify which models can use tools.

For example, with a tool-capable model like =qwen3:8b=:

- The prompt always shows: =* *qwen3:8b* ⚒ >> PROMPT:=
- The header line shows =⚒= only when you've enabled tools with C-c SPC

Additional places showing tool capability:

- The =C-c m= completing-read model selector (alongside =⊙= for vision
  models)
- The model management buffer (=C-c M=) next to model names

The list of tool-capable models is configured via
=ollama-buddy-tools-models=. Cloud model variants (e.g.
=u:qwen3-coder:480b-cloud=) are automatically matched against their base
names.

<<Overview-12>>
*** 25.1 Overview [[#Overview-12][¶]]
:PROPERTIES:
:CUSTOM_ID: overview-12
:CLASS: section
:END:
Plan mode provides a structured task planning workflow using org-mode
TODO headings. Instead of asking the LLM to do everything at once, you
describe a task and the LLM generates a multi-step plan. You can then
review, edit, skip, or execute steps individually --- with full tool
calling support.

The plan module is provided by =ollama-buddy-plan.el= and is loaded
automatically.

<<Workflow>>
*** 25.2 Workflow [[#Workflow][¶]]
:PROPERTIES:
:CUSTOM_ID: workflow
:CLASS: section
:END:
1. Type =/plan= in the prompt area to activate plan mode.
2. Describe your task and send (=C-c C-c=). The LLM generates a plan:

   #+begin_src example-preformatted
   ,** TODO Step 1: Create user model
   Define the schema with email, password hash, and timestamps.

   ,** TODO Step 2: Add registration endpoint
   Implement POST /register with input validation.

   ,** Notes
   Consider rate-limiting login attempts.
   #+end_src

3. Review the plan. Use =S-<right>=/=S-<left>= or =C-c C-t= on any step
   heading to cycle its TODO state.
4. Execute steps with =/plan-next= (one at a time) or =/plan-all= (all
   remaining). The current step is marked =DOING= and a progress
   indicator is inserted in the buffer.
5. After a step completes, use =/plan-done= to mark it =DONE=.
6. When finished, use =/plan-stop= to deactivate plan mode and restore
   your previous system prompt.

<<TODO-States>>
*** 25.3 TODO States [[#TODO-States][¶]]
:PROPERTIES:
:CUSTOM_ID: todo-states
:CLASS: section
:END:
Plan mode configures a custom org TODO sequence:

- =TODO= :: Step has not started (red).

- =DOING= :: Step is currently being executed (orange).

- =DONE= :: Step completed successfully (green). This is a “done” state
  in org terms.

- =SKIP= :: Step was skipped by the user (grey). This is also a “done”
  state.

States cycle with =S-<right>=: =TODO= → =DOING= → =DONE= → =SKIP=.

<<Slash-Commands-1>>
*** 25.4 Slash Commands [[#Slash-Commands-1][¶]]
:PROPERTIES:
:CUSTOM_ID: slash-commands-1
:CLASS: section
:END:
- =/plan= :: Activate plan mode. Sets the planning system prompt and
  configures TODO states.

- =/plan-next= :: Execute the next =TODO= step. Marks it =DOING=,
  inserts a progress indicator, switches to a minimal execution prompt,
  and sends the step to the LLM. If tools are enabled, the LLM can use
  them to carry out the work.

- =/plan-all= :: Execute all remaining =TODO= steps in sequence.

- =/plan-done= :: Mark the current =DOING= step as =DONE= and restore
  the planning prompt.

- =/plan-status= :: Show a progress summary in the echo area (e.g.
  “Plan: 2/6 done, 1 doing, 3 remaining, 0 skipped”).

- =/plan-stop= :: Deactivate plan mode. Restores the previous system
  prompt and org TODO keywords.

<<Tool-Calling-Integration>>
*** 25.5 Tool Calling Integration [[#Tool-Calling-Integration][¶]]
:PROPERTIES:
:CUSTOM_ID: tool-calling-integration
:CLASS: section
:END:
When tools are enabled (=/tools=), plan step execution uses the LLM's
native tool calling mechanism. During execution, the system prompt is
temporarily switched to a minimal one-line prompt to avoid interfering
with tool invocation. The planning prompt is restored when you mark the
step done with =/plan-done=.

<<Overview-13>>
*** 26.1 Overview [[#Overview-13][¶]]
:PROPERTIES:
:CUSTOM_ID: overview-13
:CLASS: section
:END:
RAG allows you to index local documents and source code, then retrieve
relevant context for conversations via semantic search. Instead of
pasting entire files into prompts, RAG automatically finds the most
relevant chunks and attaches them as context, keeping token usage
efficient and responses grounded in your actual documents.

The RAG module (=ollama-buddy-rag.el=) is loaded by default.

<<Requirements>>
*** 26.2 Requirements [[#Requirements][¶]]
:PROPERTIES:
:CUSTOM_ID: requirements
:CLASS: section
:END:
By default, RAG uses Ollama for embeddings and requires an embedding
model to be pulled:

#+begin_src example-preformatted
ollama pull nomic-embed-text
#+end_src

Other Ollama embedding models include =mxbai-embed-large= and
=all-minilm=. You can pull these from Model Management (C-c M) under the
“Embedding (RAG)” category.

Alternatively, you can point RAG at any OpenAI-compatible embedding
server instance) by setting =ollama-buddy-rag-embedding-base-url= and
=ollama-buddy-rag-embedding-api-style=. In that case no Ollama embedding
model is needed.

<<Quick-Start-2>>
*** 26.3 Quick Start [[#Quick-Start-2][¶]]
:PROPERTIES:
:CUSTOM_ID: quick-start
:CLASS: section
:END:
1. Pull an embedding model: =ollama pull nomic-embed-text=
2. Open the RAG menu: C-c O then r
3. Index a directory: select i and choose a directory
4. Wait for indexing to complete (progress shown in status line); press
   c to pause if needed, r to resume later
5. Search your index: select s and enter a query
6. Or attach context to a conversation: select a, enter a query, then
   chat normally
7. After editing files, update the index incrementally: select u and
   choose the same directory

<<How-It-Works-5>>
*** 26.4 How It Works [[#How-It-Works-5][¶]]
:PROPERTIES:
:CUSTOM_ID: how-it-works-5
:CLASS: section
:END:
1. *Indexing*: Files in a directory are read, split into chunks (by
   paragraph/section), and each chunk is sent to the embedding model to
   produce a vector representation. The resulting index is saved to
   disk.
2. *Searching*: Your query is embedded using the same model, then
   compared against all chunk vectors using cosine similarity. The most
   relevant chunks are returned.
3. *Attaching*: Retrieved chunks are injected into the conversation
   context as a system-level prefix, so the LLM can reference them when
   generating a response.

<<Configuration-9>>
*** 26.5 Configuration [[#Configuration-9][¶]]
:PROPERTIES:
:CUSTOM_ID: configuration-7
:CLASS: section
:END:
- =ollama-buddy-rag-embedding-model= :: The embedding model to use
  (default: ="nomic-embed-text"=).

- =ollama-buddy-rag-chunk-size= :: Target number of tokens per chunk
  (default: =300=).

- =ollama-buddy-rag-chunk-overlap= :: Number of overlapping tokens
  between consecutive chunks (default: =50=).

- =ollama-buddy-rag-top-k= :: Number of top results to return from a
  search (default: =5=).

- =ollama-buddy-rag-similarity-threshold= :: Minimum cosine similarity
  score for a chunk to be included in results (default: =0.3=).

- =ollama-buddy-rag-index-directory= :: Directory where RAG index files
  are stored (default: =~/.emacs.d/ollama-buddy-rag/=).

- =ollama-buddy-rag-file-extensions= :: List of file extensions to
  include when indexing a directory (default: common text and code
  extensions).

- =ollama-buddy-rag-exclude-patterns= :: List of glob patterns to
  exclude from indexing (default:
  =("node_modules" ".git" "__pycache__" ".elc")=).

- =ollama-buddy-rag-batch-size= :: Number of chunks to embed per API
  call (default: =10=).

- =ollama-buddy-rag-checkpoint-interval= :: Save partial progress every
  N batches during indexing (default: =5=). Set to =1= to save after
  every batch, or =0= to disable automatic checkpointing (progress is
  still saved on pause).

- =ollama-buddy-rag-embedding-base-url= :: Base URL for the embedding
  service. When =nil= (the default), the main Ollama host and port are
  used. Set this to point at a separate OpenAI-compatible embedding
  server, for example:

  #+begin_src example-preformatted
  (setq ollama-buddy-rag-embedding-base-url "http://localhost:9000")
  #+end_src

  This is useful when running a dedicated embedding service
  independently from your Ollama instance.

- =ollama-buddy-rag-embedding-api-style= :: The API style used by the
  embedding endpoint. Two values are supported:

  - =ollama= :: (default) POST to =/api/embed= --- the Ollama native
    endpoint.

  - =openai= :: POST to =/v1/embeddings= --- compatible with any
    OpenAI-style embedding service. Use this together with
    =ollama-buddy-rag-embedding-base-url=.

- =ollama-buddy-rag-embedding-api-key= :: Optional Bearer token sent to
  the embedding service. Only needed when
  =ollama-buddy-rag-embedding-api-style= is =openai= and the service

<<PDF-Support>>
*** 26.6 PDF Support [[#PDF-Support][¶]]
:PROPERTIES:
:CUSTOM_ID: pdf-support
:CLASS: section
:END:
RAG can index PDF files if =pdftotext= (from =poppler-utils=) is
installed on your system. PDF support is auto-detected at runtime:

- If =pdftotext= is available, =.pdf= files are included when indexing
  directories.
- If =pdftotext= is not installed, PDF files are silently skipped.
- PDFs bypass the file size limit (=ollama-buddy-rag-max-file-size=)
  since binary PDF size does not reflect extracted text size.

To install =pdftotext=:

#+begin_src example-preformatted
# Debian/Ubuntu
sudo apt install poppler-utils

# Arch Linux
sudo pacman -S poppler

# macOS
brew install poppler
#+end_src

<<Interactive-Commands-1>>
*** 26.7 Interactive Commands [[#Interactive-Commands-1][¶]]
:PROPERTIES:
:CUSTOM_ID: interactive-commands-1
:CLASS: section
:END:
These commands are available from the RAG transient menu (C-c O then r):

- =ollama-buddy-rag-index-or-update-directory (==i==)= :: Index or
  update a directory. If an index already exists for the directory, it
  is incrementally updated: only files that have been added or modified
  since the last indexing are re-chunked and re-embedded, chunks from
  unchanged files are retained, and chunks for deleted files are
  removed. If no existing index is found, a full index is created. If
  the embedding model has changed since the index was created, a full
  re-index is required and the user is prompted before proceeding.

- =ollama-buddy-rag-pause (==c==)= :: Pause an in-progress indexing or
  update operation and save partial progress to disk. The pause takes
  effect before the next embedding batch is sent, so the current
  in-flight batch completes but no further batches are processed. The
  partial index is saved with all already-embedded chunks and can be
  resumed later. =ollama-buddy-rag-cancel= is an alias for this command.

- =ollama-buddy-rag-resume (==r==)= :: Resume a previously paused
  indexing operation. Loads the partial index, retains all
  already-embedded chunks, and continues embedding the remaining pending
  chunks. If only one partial index exists it is selected automatically;
  otherwise you are prompted to choose. The embedding model must match
  the one used when the index was created.

- =ollama-buddy-rag-search (==s==)= :: Search the index with a text
  query. Results are displayed in an org-mode buffer showing similarity
  scores, file paths, and content excerpts.

- =ollama-buddy-rag-attach (==a==)= :: Search and attach the top results
  as context for the next conversation. Attached RAG context appears
  when you use Show Attachments.

- =ollama-buddy-rag-list-indexes (==w==)= :: List all saved RAG indexes
  with their chunk counts, file counts, and whether the index supports
  incremental updates.

- =ollama-buddy-rag-delete-index (==d==)= :: Delete a saved RAG index.

<<Incremental-Updates>>
*** 26.8 Incremental Updates [[#Incremental-Updates][¶]]
:PROPERTIES:
:CUSTOM_ID: incremental-updates
:CLASS: section
:END:
When you first index a directory with i, the index records modification
times for every file. On subsequent runs you can use u (Update
Directory) instead of re-indexing from scratch:

1. Files whose modification time has changed are re-chunked and
   re-embedded.
2. New files are chunked and embedded.
3. Chunks belonging to deleted files are removed.
4. Chunks for unchanged files are kept as-is --- no embedding calls are
   made for them.

This makes updates significantly faster for large directories where only
a few files have changed.

Incremental updates require the same embedding model that was used to
create the index. If you change =ollama-buddy-rag-embedding-model=, a
full re-index is needed (you will be prompted).

Indexes created before version 3.6.0 do not contain file metadata.
Running Index/Update Directory on such an index automatically falls back
to a full re-index and saves the metadata for future incremental
updates.

<<Pause-and-Resume>>
*** 26.9 Pause and Resume [[#Pause-and-Resume][¶]]
:PROPERTIES:
:CUSTOM_ID: pause-and-resume
:CLASS: section
:END:
Indexing large directories can take a long time. If you need to close
Emacs or stop the process, you can pause and resume later.

- *Pause* :: Press c in the RAG transient menu (or run
  =M-x ollama-buddy-rag-pause=). The current batch completes, then all
  progress is saved to disk as a partial index. The partial index is
  immediately searchable for any chunks that have already been embedded.

- *Resume* :: Press r in the RAG transient menu (or run
  =M-x ollama-buddy-rag-resume=). The partial index is loaded and
  embedding continues from where it stopped. When all remaining chunks
  are processed, the index is saved as a complete index.

- *Automatic checkpointing* :: Progress is also saved periodically
  during indexing (every 5 batches by default, controlled by
  =ollama-buddy-rag-checkpoint-interval=). This means even an unexpected
  Emacs crash preserves most of the work done so far. The next
  =ollama-buddy-rag-resume= picks up from the last checkpoint.

- *Partial index visibility* :: =M-x ollama-buddy-rag-list-indexes=
  marks partial indexes with =[partial: N/M chunks]= in the heading and
  shows a count of pending chunks.

- *Partial index handling* :: Running
  =M-x ollama-buddy-rag-index-or-update-directory= on a directory with a
  partial index will offer to resume it or start a fresh full re-index.

<<Inline-_0040rag_0028_0029-Syntax>>
*** 26.10 Inline @rag() Syntax [[#Inline-_0040rag_0028_0029-Syntax][¶]]
:PROPERTIES:
:CUSTOM_ID: inline-rag-syntax
:CLASS: section
:END:
You can use =@rag(query)= directly in prompts to automatically search
your RAG index and attach results as context --- no separate attach step
needed. This mirrors the =@search(query)= syntax for web search.

#+begin_src example-preformatted
Tell me about @rag(giant squid attack) from the book.
#+end_src

Multiple queries are supported:

#+begin_src example-preformatted
Compare @rag(captain nemo) with @rag(ned land) in terms of character development.
#+end_src

How it works:

1. The =@rag()= patterns are extracted from the prompt.
2. Each query is embedded synchronously and searched against the index.
3. Matching chunks are attached as RAG context.
4. The =@rag()= wrappers are removed, preserving the query text in the
   prompt sent to the model.

If you have multiple indexes, you will be prompted to select one. If
only one index exists, it is used automatically.

<<Example-Workflow>>
*** 26.11 Example Workflow [[#Example-Workflow][¶]]
:PROPERTIES:
:CUSTOM_ID: example-workflow
:CLASS: section
:END:
Suppose you have the text of “Twenty Thousand Leagues Under the Sea” in
=~/books/leagues.txt=:

#+begin_src example-preformatted
;; 1. Index the book
C-c O r i  ->  ~/books/

;; 2a. Attach via transient menu
C-c O r a  ->  "giant squid attack"

;; 2b. Or use inline syntax -- no separate step needed
"Describe the @rag(giant squid encounter) in detail, referencing the text."

;; 3. Ask a question in chat -- the LLM now has relevant passages
"Describe the giant squid encounter in detail, referencing the text."
#+end_src

Without RAG, the model would give a generic answer from training data.
With RAG, it references specific passages from your indexed document.

<<Managing-Token-Usage>>
*** 27.1 Managing Token Usage [[#Managing-Token-Usage][¶]]
:PROPERTIES:
:CUSTOM_ID: managing-token-usage
:CLASS: section
:END:
Ollama Buddy tracks token usage on every response via org property
drawers. Each =** [model: RESPONSE]= heading receives a =:PROPERTIES:=
drawer with =:TIMESTAMP:=, =:TOKENS:=, =:RATE:=, =:ELAPSED:=, and
=:WAIT:= (time to first token). The drawer is folded by default; press
=TAB= on the heading to reveal it.

#+begin_src example-preformatted
,** [qwen3:8b: RESPONSE]
:PROPERTIES:
:TIMESTAMP: [2026-03-16 Sun 14:32]
:TOKENS:   142
:RATE:     23.4
:ELAPSED:  6.1s
:WAIT:     2.1s
:END:
#+end_src

You can use =org-columns= to get a tabular overview of all responses in
a session. The data survives session save/load since it is in the buffer
text.

To view aggregate token usage statistics:

#+begin_src example-preformatted
M-x ollama-buddy-display-token-stats
#+end_src

or press =C-c #= in the chat buffer.

This displays a combined view with token usage graphs by model, average
token rates, and recent interactions.

<<Connection-Problems>>
**** 30.1.1 Connection Problems [[#Connection-Problems][¶]]
:PROPERTIES:
:CUSTOM_ID: connection-problems
:CLASS: subsection
:END:
- Symptom: Unable to connect to Ollama server :: - Check if Ollama is
    running with =ps aux | grep ollama=
  - Verify host and port settings (=ollama-buddy-host= and
    =ollama-buddy-port=)
  - Try connecting to Ollama directly:
    =curl http://localhost:11434/api/tags=
- Symptom: Connection breaks during long responses :: - This can happen
    with very large responses
  - Try setting a lower =num_predict= value
  - Check if your OS has any network timeout settings

<<Model-Problems>>
**** 30.1.2 Model Problems [[#Model-Problems][¶]]
:PROPERTIES:
:CUSTOM_ID: model-problems
:CLASS: subsection
:END:
- Symptom: Model loads but gives poor responses :: - Try a different
    model
  - Adjust parameters (increase temperature for more creativity)
  - Provide clearer or more detailed prompts
  - Check if the model is appropriate for your task
- Symptom: Model fails to load or crashes :: - Check system memory usage
  - Try a smaller quantized model
  - Adjust =num_ctx= to a smaller value
  - Set =low_vram= to =t= if using GPU

<<Interface-Issues>>
**** 30.1.3 Interface Issues [[#Interface-Issues][¶]]
:PROPERTIES:
:CUSTOM_ID: interface-issues
:CLASS: subsection
:END:
- Symptom: Chat buffer becomes unresponsive :: - Cancel any running
    requests with =C-c k=
  - Check if Emacs is using high CPU
  - Try disabling token statistics display
  - Close and reopen the chat buffer
- Symptom: Markdown conversion issues :: - Toggle markdown conversion
    off with =C-c C-o=
  - Check if the response contains complex formatting
  - Try editing the history to fix formatting issues

<<Report-Issues>>
**** 30.2.3 Report Issues [[#Report-Issues][¶]]
:PROPERTIES:
:CUSTOM_ID: report-issues
:CLASS: subsection
:END:
If you encounter a bug:

1. Enable debug mode
2. Reproduce the issue
3. Copy the debug output
4. Report the issue on GitHub with:
   - Emacs version
   - Ollama version
   - Model used
   - Debug output
   - Steps to reproduce

*** 31.1 Getting Started [[#Getting-Started][¶]]
:PROPERTIES:
:CUSTOM_ID: getting-started
:CLASS: section
:END:
Ollama Buddy is an open-source project, and contributions are welcome!

1. Fork the repository:
   [[https://github.com/captainflasmr/ollama-buddy]]
2. Clone your fork:
   =git clone https://github.com/YOUR-USERNAME/ollama-buddy.git=
3. Create a branch: =git checkout -b my-feature-branch=
4. Make your changes
5. Test thoroughly
6. Commit with a clear message
7. Push to your fork
8. Create a pull request

<<User-System-Prompts-Issues>>
**** 31.5.1 User System Prompts Issues [[#User-System-Prompts-Issues][¶]]
:PROPERTIES:
:CUSTOM_ID: user-system-prompts-issues
:CLASS: subsection
:END:
- Symptom: Cannot save system prompts :: - Check if
    ‘ollama-buddy-user-prompts-directory‘ exists and is writable
  - Ensure you have set a system prompt before trying to save it
  - Check for any error messages in the minibuffer or *Messages* buffer
- Symptom: Prompts not appearing in the load menu :: - Verify that
    prompts are saved in the correct format
    (category__title__system.org)
  - Check if the prompts directory contains files with proper formatting
  - Try refreshing the prompts cache with ‘M-x
    ollama-buddy-user-prompts--refresh-cache‘
