1 Introduction ¶

  • Why Use Ollama Buddy?
  • Prerequisites

1.1 What is Ollama Buddy? ¶

Ollama Buddy is an Emacs package that provides a friendly AI assistant interface to Ollama, a tool for running large language models (LLMs) locally on your computer. It allows you to interact with AI models directly from within Emacs for various tasks such as:

  • Code refactoring and explanation
  • Writing assistance and proofreading
  • Generating Git commit messages
  • Dictionary lookups and language assistance
  • Custom AI-powered workflows via roles
  • Integrating with commercial APIs (OpenAI, Claude, Gemini, Grok, Copilot, Codestral, DeepSeek, OpenRouter, OpenCode Go)
  • Using Ollama cloud models
  • Including files as context in conversations
  • Real-time web search to provide current information to LLMs
  • Tool calling — enabling LLMs to read files, run commands, search buffers and more

Instead of context-switching to web interfaces or terminal applications, Ollama Buddy brings the power of local LLMs right into your Emacs workflow.

1.2 Why Use Ollama Buddy? ¶

  • Privacy: All interactions happen locally with Ollama - no data sent to external services unless you use commercial APIs
  • Integration: Seamlessly fits into your existing Emacs workflow
  • Flexibility: Supports multiple models, parameter tuning, and custom commands
  • Efficiency: Quick access to AI assistance without leaving your editor
  • Extensibility: Create custom roles and commands for your specific needs
  • File Support: Include text files, code, and documentation directly in conversations

1.3 Prerequisites ¶

Before using Ollama Buddy, you need:

  • Emacs 29.1 or later
  • Ollama installed and running on your system (see https://ollama.ai)
  • At least one language model pulled into Ollama
  • (Optional) API keys for remote providers (OpenAI, Claude, Gemini, Grok, Copilot, Codestral, DeepSeek, OpenRouter, OpenCode Go) if you want to use those services

2.1 Installing Ollama ¶

Before installing Ollama Buddy, you need to install Ollama itself:

  1. Visit https://ollama.ai and download the installer for your platform
  2. Install and run Ollama according to the instructions
  3. Pull at least one model using ollama pull llama3:latest (or another model of your choice)

2.2 Package Installation ¶

  • Using package.el
  • Using use-package
  • Manual Installation

2.2.1 Using package.el ¶

The recommended way to install Ollama Buddy is through MELPA: 

M-x package-install RET ollama-buddy RET

2.2.2 Using use-package ¶

If you use use-package, add the following to your Emacs configuration: 

(use-package ollama-buddy
  :ensure t
  :bind ("C-c o" . ollama-buddy-role-transient-menu))

With a default model: 

(use-package ollama-buddy
  :ensure t
  :bind ("C-c o" . ollama-buddy-role-transient-menu)
  :custom (ollama-buddy-default-model "llama3:latest"))

2.2.3 Manual Installation ¶

To install manually:

  1. Clone the repository: 
    git clone https://github.com/captainflasmr/ollama-buddy.git
  2. Add to your configuration: 
    (add-to-list 'load-path "/path/to/ollama-buddy")
(require 'ollama-buddy)
(global-set-key (kbd "C-c o") #'ollama-buddy-role-transient-menu)

2.3 Dependencies ¶

Ollama Buddy requires the following:

  • Emacs 29.1 or later
  • (Optional) transient 0.4.0+ for advanced menu system

Built-in packages used: json, cl-lib, url, subr-x, dired, org, savehist.

2.4 Communication Backends ¶

Ollama Buddy supports two communication backends:

network-process (default)

Uses Emacs built-in network process.

curl

Uses external curl command for requests. Useful as a fallback when network process has issues.

To switch backends: 

M-x ollama-buddy-switch-communication-backend

or press C-c e in the chat buffer. Also available from the transient menu (C-c O, e) and via the /backend slash command.

When the curl backend is active, the header line shows a ⇄ indicator.

To configure curl backend: 

(require 'ollama-buddy-curl)
(setq ollama-buddy-communication-backend 'curl)
(setq ollama-buddy-curl-executable "curl")
(setq ollama-buddy-curl-timeout 300)

2.5 API Key Setup ¶

If you want to use OpenAI or Claude integration, you’ll need to set up API keys securely:

  1. Use Emacs built-in auth-source for secure storage
  2. Add to your auth sources (e.g., ~/.authinfo.gpg): 
    machine api.openai.com login apikey password YOUR_OPENAI_API_KEY_HERE
machine api.anthropic.com login apikey password YOUR_CLAUDE_API_KEY_HERE
  3. Alternatively, set the variables directly (less secure): 
    (setq ollama-buddy-openai-api-key "your-openai-key")
(setq ollama-buddy-claude-api-key "your-claude-key")

3.7.1 Provider Prefixes ¶

Prefixes are only used when external providers (OpenAI, Claude, etc.) are loaded:

When only Ollama is used

Models appear without prefix: llama3:8b, qwen3-coder:480b-cloud

When external providers are loaded

Prefixes disambiguate providers:

  • o: - Ollama local (e.g., o:llama3:8b)
  • u: - Ollama cloud (e.g., u:qwen3-coder:480b-cloud)
  • a: - OpenAI (e.g., a:gpt-4)
  • c: - Claude (e.g., c:claude-3-opus)
  • g: - Gemini (e.g., g:gemini-pro)
  • k: - Grok (e.g., k:grok-1)
  • p: - GitHub Copilot (e.g., p:gpt-4o)
  • s: - Codestral (e.g., s:codestral-latest)

The ☁ cloud symbol provides visual indication for cloud models regardless of whether prefixes are used.

4 Quick Start ¶

  • Basic Usage
  • Common Operations

4.1 Basic Usage ¶

  1. Launch Ollama Buddy: 
    M-x ollama-buddy-role-transient-menu

    or use your configured keybinding (e.g., C-c o).

  2. The menu will show available options. Press the corresponding key for the action you want.
  3. To open the chat interface, press o or select "Open Chat".
  4. In the chat buffer, type your prompt and press C-c C-c to send it.
  5. The AI will respond in the chat buffer.

4.2 Common Operations ¶

Sending text from a file

Select text in any buffer, then press C-c o and choose "Send Region" (or press l).

Refactoring code

Select code, press C-c o, then choose "Refactor Code" (or press r).

Generating a commit message

Select your changes, press C-c o, then choose "Git Commit Message" (or press g).

C-c m

Change model

C-c l

Pull new model Press C-c m to switch between available models. Press C-c l to pull new models from Ollama Hub. Cloud models appear in the list marked with ☁.

Using the transient menu

Press C-c O for the main transient menu with all features organized by category.

Attaching files

Press C-c C-a to attach a file, or use the transient menu “Project” section.

Project Integration

Press C-c P for the Project sub-menu. Access project-specific file attachment and context.

Peeking at thinking blocks

While a model is thinking, the *** Thinking heading is folded. Press TAB on the heading to peek at accumulated content; press TAB again to re-fold.

Toggling Org/Markdown conversion

Press C-c C-o to switch between Org-mode and Markdown output. Also available from the transient menu under “Settings”.

Toggling Global System Prompt

Press C-c < to toggle the global system prompt on or off. The header line shows < when disabled.

6.2 Interface Overview ¶

The chat interface consists of:

  • A welcome message with available models and providers
  • Conversation history (previous prompts and responses)
  • A prompt area for entering your queries with model capability indicators
  • A header line with status information including:
    • Cloud model indicator (☁) - using an Ollama cloud model
    • Tool calling enabled indicator (⚒) - tools enabled AND model supports them
    • Vision indicator (⊙) - model supports vision
    • Attachment indicators (≡N)
    • Web search indicators (♁N)
    • Context usage pie chart (GUI) or bar (terminal), when enabled
    • Global system prompt disabled indicator (<)
    • History count (H5/10, off by default)
    • Current model name
    • System prompt indicator
    • Modified parameters
  • A prompt line showing model capabilities:

    • ⚒ when model supports tool calling (regardless of enabled state)
    • ✦ when model supports thinking/reasoning
  • Context warnings and validation

6.3 Sending Prompts ¶

To send a prompt to the AI:

  1. Type your message in the prompt area (after ">> PROMPT:")
  2. Press C-c C-c to send
  3. Wait for the AI to generate a response

You can also:

  • Use C-u C-c C-c (or C-u C-c RET) to send “continue” without typing anything — useful when a model stops mid-response or after tool results
  • Use C-u C-u C-c C-c to rewind the conversation to a previous prompt
  • Use M-p and M-n to navigate through prompt history
  • Press C-c k to cancel a request if it’s taking too long
  • Press C-a for smart beginning-of-line: first press jumps to the start of the prompt text (after “>> PROMPT:”), second press goes to column 0

8.1 Understanding Context Windows ¶

Context windows define how much text (measured in tokens) a model can process at once. This includes your current prompt, conversation history, any system prompts, and attached files. Understanding and managing context is crucial for:

  • Preventing errors when context limits are exceeded
  • Optimizing model performance for different tasks
  • Managing longer conversations efficiently
  • Including files without exceeding context limits

8.2 Context Size Detection ¶

Ollama Buddy uses multiple methods to determine a model’s context size:

  1. Built-in mappings for popular models (llama3, mistral, codellama, etc.)
  2. Custom context sizes set via the num_ctx parameter
  3. Manual configuration through interactive commands
  4. Fallback to reasonable defaults (4096 tokens) for unknown models

8.3 Enabling Context Monitoring ¶

Context monitoring is disabled by default. To enable it: 

(setq ollama-buddy-show-context-percentage t)

With context monitoring enabled:

  • The status bar shows current/max context usage (e.g., "2048/8192")
  • Text formatting indicates usage levels:
    • Normal font: Under 85% usage
    • Bold and underlined: 85-100% usage
    • Inverted: At or exceeding 100% usage
  • Warnings appear before sending prompts that exceed limits

8.4 Context with File Attachments ¶

File attachments are included in context calculations:

  • Each attached file contributes to the total token count
  • The context breakdown shows attachment tokens separately
  • File content is included in the request context
  • Large files can significantly impact context usage

8.5 Context Management Commands ¶

Show Context Information (C-c C)

Displays a breakdown of current context usage with a pie chart (GUI Emacs) or a composite text bar (terminal Emacs). Each segment represents a context category (history, system prompt, attachments, web search, RAG, current prompt, free space) with a colour-coded legend showing token counts and percentages.

Set Model Context Size (C-c $)

Manually configure the context size for a specific model.

Toggle Context Display (C-c %)

Show or hide the context percentage in the status bar.

8.6 Token Estimation ¶

Ollama Buddy estimates token counts using a heuristic approach:

  • Each word is multiplied by 1.3 (following common approximations)
  • This provides a reasonable estimate for most use cases
  • Actual token counts may vary slightly between models

8.7.1.1 Paste-and-Send Approach ¶

  1. Paste your content into the chat buffer
  2. Press the send keybinding
  3. If context is exceeded, you’ll get a warning dialog
  4. Choose whether to proceed or modify your content

  3. If too high:
    • Trim your current prompt
    • Edit conversation history (C-c H)
    • Switch to a larger context model

The num_ctx parameter allows you to set a specific context size:

  2. Select num_ctx
  3. Enter your desired context size
  4. Ollama Buddy will respect this limit

8.10 Troubleshooting Context Issues ¶

Context warnings appear unexpectedly
  • Check if you have a long system prompt
  • Review conversation history length
  • Verify the model’s actual context size
  • Check if files are attached and their sizes
Model responses are truncated
  • Increase the num_ctx parameter
  • Reduce history length with C-c Y
  • Clear some conversation history
  • Remove large file attachments
Context calculations seem inaccurate
  • Remember that token estimation is approximate
  • Different models may tokenize text differently
  • Use C-c C to see detailed breakdowns

10.1 Overview ¶

Web search integration allows Ollama Buddy to fetch real-time information from the web and include it as context for your LLM conversations. This is particularly useful for:

  • Getting current information beyond the model’s training cutoff
  • Researching topics and summarizing findings
  • Fact-checking with up-to-date sources

10.2 How It Works ¶

The web search feature implements a multi-stage pipeline. Content can be retrieved using two methods, controlled by ollama-buddy-web-search-content-source:

  • eww Mode (Default, Recommended)
  • API Mode (Experimental)

10.2.1 eww Mode (Default, Recommended) ¶

When ollama-buddy-web-search-content-source is set to 'eww (the default):

  1. Query to Ollama API: Search queries are sent to Ollama’s Web Search API (https://ollama.com/api/web_search) via REST with Bearer token authentication.
  2. URL Extraction: The API returns search results containing URLs.
  3. eww/shr Processing: Each URL is fetched and processed through Emacs’ built-in eww/shr HTML renderer:
    • HTML is parsed using libxml-parse-html-region
    • Main content is extracted (looking for <article>, <main>, or content divs)
    • Content is rendered to clean plain text using shr-insert-document
    • Org-mode special characters are escaped (* and # at line starts become ,* and ,#)
  4. Context Attachment: The cleaned text is formatted with org headings and attached to the conversation context.
  5. LLM Submission: The search results are included when sending prompts to any configured LLM provider.

This mode produces cleaner, more complete content but requires additional HTTP requests to fetch each URL.

10.2.2 API Mode (Experimental) ¶

When ollama-buddy-web-search-content-source is set to 'api:

  1. Query to Ollama API: Search queries are sent to the Web Search API.
  2. Direct Content: Content snippets returned directly by the Ollama API are used without fetching individual URLs.
  3. Context Attachment: The API-provided content is formatted and attached to the conversation context.

This mode is faster (no additional HTTP requests) but the content quality depends on what the Ollama API returns, which may be less refined than eww-rendered pages. Use this mode when speed is more important than content completeness.

10.3 Configuration ¶

To use web search, you need an API key from Ollama:

  1. Visit https://ollama.com/settings/keys
  2. Create a new API key
  3. Configure in Emacs: 
;; Required: Set your API key
(setq ollama-buddy-web-search-api-key "your-api-key-here")

;; Optional customizations
(setq ollama-buddy-web-search-max-results 5)        ; Number of URLs to fetch (default: 5)
(setq ollama-buddy-web-search-snippet-length 2000)  ; Max chars per result (default: 500)
(setq ollama-buddy-web-search-include-urls nil)     ; Include URLs in context (default: nil)
(setq ollama-buddy-web-search-content-source 'eww)  ; Content source: 'eww (default) or 'api

Note: The web search module is now automatically loaded with Ollama Buddy. You no longer need to explicitly (require 'ollama-buddy-web-search).

12.7 Session Naming ¶

When saving a session, Ollama Buddy generates a default name from the first user message. The name is created by filtering out common English stop words (“the”, “a”, “is”, etc.) and short words (less than 3 characters), then joining up to 5 remaining key terms with hyphens. A timestamp prefix is always prepended for uniqueness and chronological sorting.

For example, a conversation starting with “Tell me about the history of space exploration” would generate the session name 2026-03-07-170800-history-space-exploration.

12.8 Renaming a Session ¶

Use /rename in the prompt area to set a display name for the current session. If no name has been set, an auto-generated one from the first prompt is offered for editing. If a name already exists, it is shown for editing.

The display name is used as the descriptive suffix when saving—/save always prepends a fresh timestamp, so renaming and saving multiple times produces distinct files without duplicate timestamps.

12.9 Session Directory Persistence ¶

Sessions save the chat buffer’s default-directory. When a session is loaded, the directory is restored (provided it still exists on disk). This means loading a session that was started in a project automatically restores the project context, so project.el integration, project summary auto-loading, and file-relative commands all work correctly.

12.10 Autosave Transcript ¶

Ollama Buddy automatically saves the chat buffer to a recovery file after each completed response. The file is saved as ~autosave.org in the sessions directory (ollama-buddy-sessions-directory).

This provides crash protection — if Emacs exits unexpectedly, you can recover the conversation from the autosave file.

The autosave file is automatically deleted when you perform a proper session save with C-c S.

12.11 Conversation History ¶

Sessions save the conversation history for each model separately.

M-x ollama-buddy-history-edit

or press C-c H in the chat buffer.

M-x ollama-buddy-clear-history

or press C-c X in the chat buffer.

To toggle whether history is used in requests: 

M-x ollama-buddy-toggle-history

or press C-c J in the chat buffer.

12.12 Rewinding a Conversation ¶

You can rewind a conversation to a previous prompt in several ways:

  • C-u C-u C-c C-c — when point is on a prompt heading, rewinds directly to that prompt. Otherwise falls back to completing-read.
  • /rewind — slash command that always presents completing-read.
  • C-c O ⇒ u — transient menu “Rewind”, always presents completing-read.

When using C-u C-u C-c C-c on a heading, you can navigate there with:

  • Fold the buffer to show only prompt headings (S-TAB to cycle global visibility).
  • Use imenu to jump to a specific prompt.
  • Simply scroll up and place point on a ‘>> PROMPT:’ heading.

After confirmation, everything from that prompt onward is removed and the conversation history is truncated to match. The prompt text is pre-filled in a fresh prompt area so you can edit and resend it.

14.2 Role File Naming Convention ¶

The file naming convention is critical to understand how roles, preset files, and menu configurations work together:

Required filename format

ollama-buddy--preset__ROLE-NAME.el

  • The double underscore __ separates the prefix from your role name
  • The role name portion becomes the identifier shown when switching roles
  • Example: ollama-buddy--preset__programmer.el creates a role named "programmer"

This naming convention is how Ollama Buddy discovers and identifies role files. When you run ollama-buddy-roles-switch-role, the system:

  1. Scans the ollama-buddy-roles-directory for files matching the pattern
  2. Extracts the role name from each filename (the part after __)
  3. Presents these names in the role selection interface
  4. When selected, loads the corresponding file which redefines ollama-buddy-command-definitions
  5. This redefinition immediately changes the available commands in your Ollama Buddy menu

The relationship chain works like this: 

ollama-buddy--preset__ROLE-NAME.el → Defines ollama-buddy-command-definitions → Controls menu content

When creating roles using the interactive role creator (C-c E), this naming convention is automatically handled for you. When creating roles manually, you must follow this pattern for Ollama Buddy to recognize your role files correctly.

  1. Create a file named ollama-buddy--preset__your-role-name.el in your ollama-buddy-roles-directory
  2. Structure your file like this: 
;; ollama-buddy preset for role: programmer
(require 'ollama-buddy)

(setq ollama-buddy-command-definitions
  '(
    ;; Standard commands
    (open-chat
     :key ?o
     :description "Open chat buffer"
     :group "General"
     :action ollama-buddy--open-chat)

    (switch-role
     :key ?R
     :description "Switch roles"
     :group "General"
     :action ollama-buddy-roles-switch-role)

    ;; Custom commands for this role
    (refactor-code
     :key ?r
     :description "Refactor code"
     :group "Code"
     :model "codellama:7b"
     :prompt "Refactor this code to improve readability and efficiency:"
     :system "You are an expert software engineer who improves code quality."
     :action (lambda () (ollama-buddy--send-with-command 'refactor-code))
     :destination in-buffer)

    (explain-code
     :key ?e
     :description "Explain code"
     :group "Code"
     :model "deepseek-r1:7b"
     :prompt "Explain what this code does in detail:"
     :system "You are a programming teacher who explains code clearly."
     :action (lambda () (ollama-buddy--send-with-command 'explain-code))
     :destination chat)

    (git-commit
     :key ?g
     :description "Git commit message"
     :group "Utility"
     :prompt "Write a concise git commit message for these changes:"
     :system "You are a version control expert who creates clear commit messages."
     :action (lambda () (ollama-buddy--send-with-command 'git-commit))
     :destination chat)

    ;; System commands
    (custom-prompt
     :key ?e
     :description "Custom prompt"
     :group "System"
     :action ollama-buddy--menu-custom-prompt)
    ))

The :group property is optional. Commands sharing the same :group value appear together in a named column in the transient role menu (ollama-buddy-role-transient-menu). Presets without any :group properties still work — all commands appear in a single “Commands” column for full backwards compatibility.

14.5 Command Grouping ¶

The :group property on command definitions controls how commands are organized into columns in the transient role menu. This provides a clean, discoverable layout: 

;; Commands with :group "Code" appear in a "Code" column
(refactor-code
 :key ?r
 :description "Refactor code"
 :group "Code"
 :action ...)

(explain-code
 :key ?e
 :description "Explain code"
 :group "Code"
 :action ...)

Built-in presets use groups like “General”, “Code Analysis”, “Code Generation”, “Custom”, “System”, etc. You can use any group names you like in your own presets.

14.6 Per-Command Response Destination ¶

The optional :destination property on a command definition overrides the global ollama-buddy-in-buffer-replace toggle for that specific command:

in-buffer

Always stream the response back into the source buffer, replacing the selected region (requires ollama-buddy-rewrite to be loaded). The global toggle is ignored.

chat

Always open the chat buffer, regardless of the global toggle.

nil (or absent)

Honour ollama-buddy-in-buffer-replace as before — this is the default and preserves full backwards compatibility.

Use :destination in-buffer for commands that transform selected text (refactoring, proofreading, translation, style rewrites) and :destination chat for commands that analyse or generate new content about the selection (code explanations, commit messages, dictionary lookups). 

;; Transforms selected text --- always rewrites in-buffer
(refactor-code
 :key ?r
 :description "Refactor code"
 :prompt "Refactor this code:"
 :action (lambda () (ollama-buddy--send-with-command 'refactor-code))
 :destination in-buffer)

;; Generates new content --- always goes to chat
(explain-code
 :key ?e
 :description "Explain code"
 :prompt "Explain this code:"
 :action (lambda () (ollama-buddy--send-with-command 'explain-code))
 :destination chat)

;; No :destination --- honours the global toggle
(describe-code
 :key ?c
 :description "Describe code"
 :prompt "Describe this code:"
 :action (lambda () (ollama-buddy--send-with-command 'describe-code)))

All built-in presets ship with :destination annotations. Commands added via ollama-buddy-update-menu-entry or add-to-list on ollama-buddy-command-definitions work identically — simply include or omit the :destination key as appropriate.

17.1 Overview ¶

The /launch slash command lets you launch any Ollama model (local or cloud) in an external terminal with a frontend tool such as Claude Code.

There are two types of agents:

Standard agents

Use ollama launch FRONTEND --model MODEL. These are built-in Ollama integrations (Claude Code, Codex, Pi, etc.).

Direct agents

Invoked as standalone executables in a terminal, bypassing ollama launch entirely. This allows external tools that are not built-in Ollama integrations to participate in the launch workflow. Direct agents only require their own executable on PATH — they do not depend on ollama being installed.

17.2 Usage ¶

Slash command

Type /launch in the chat prompt area. You will be prompted to select a model, then a frontend (if more than one is configured).

Model Management

Each model in the C-c M Model Management buffer has a Launch button (when frontends are detected on PATH).

Interactive (from chat)

M-x ollama-buddy-launch

Standalone (from anywhere)

M-x ollama-buddy-launch-external or press v in the main transient menu (C-c O). This does not require the Ollama Buddy chat buffer — it launches an agent directly from the current directory. In Dired, the terminal opens in the displayed directory. The model prompt shows the target directory for confirmation.

All launch methods produce fully detached terminal sessions that survive closing Emacs.

17.3 Terminal Auto-Detection ¶

When ollama-buddy-launch-terminal is nil (the default), the terminal emulator is auto-detected in this order:

  1. TERMINAL environment variable
  2. TERM_PROGRAM environment variable (set by modern terminals)
  3. xdg-terminal-exec (freedesktop.org standard)
  4. PATH scan of common terminals: kitty, alacritty, foot, wezterm, ghostty, gnome-terminal, kgx, konsole, xfce4-terminal, mate-terminal, lxterminal, tilix, terminator, st, urxvt, xterm

Each terminal is paired with its correct command-line flag (-e for most, -- for gnome-terminal, start -- for wezterm).

17.4 Welcome Screen ¶

When frontends are detected, the welcome screen shows an indicator: 

⚡ launch: claude, omp* (via kitty)

Direct agents are marked with * to distinguish them from standard ollama launch integrations.

17.5 Configuration ¶

ollama-buddy-launch-terminal

Terminal emulator to use. When nil, auto-detects.

ollama-buddy-launch-terminal-flag

Flag to pass a command to the terminal. When nil, auto-detects. Most terminals use -e; gnome-terminal uses --.

ollama-buddy-launch-extra-agents

Additional agents beyond the built-in registry. Each entry is a plist (:name :executable :label) plus optional keys :direct and :model-flag. Entries here override built-in agents with the same :name. Use :direct t for tools that are not ollama launch integrations.

ollama-buddy-launch-small-model-threshold

Minimum recommended parameter count (in billions) for agent launch (default: 7). Models below this threshold trigger a confirmation warning. Set to nil to disable the warning.

17.6 Small Model Warning ¶

Coding agents such as Claude Code, OpenCode, Codex, and others prepend large system prompts (20,000+ tokens) to every request. These prompts contain tool definitions, safety guardrails, formatting instructions, and other context that the agent needs to function.

This means that even a simple message like “hello” actually requires the model to process over 20,000 tokens before it can generate a single word of output. On small models (below \sim7B parameters), especially without a dedicated GPU, this can cause:

  • The model appearing completely frozen for minutes at a time
  • No visible output while the model slowly encodes the system prompt
  • Extremely slow responses once generation finally begins
  • Token counts showing zero in the agent UI while prompt encoding is still in progress

Recommendations ¶

  • Use models with 7B+ parameters for agent workflows (14B+ preferred)
  • A dedicated GPU with sufficient VRAM is strongly recommended — CPU inference with large agent contexts is impractical for interactive use
  • Consider Ollama cloud models (e.g. qwen3-coder:480b-cloud) which offload compute to Ollama’s infrastructure
  • If a model appears frozen after launch, run ollama ps in a terminal to see whether the model is loaded and how much is offloaded to CPU

When you launch a model below the threshold set in ollama-buddy-launch-small-model-threshold (default 7B), Ollama Buddy will display a confirmation warning explaining the potential performance impact. This warning is skipped for cloud models, which run on remote infrastructure.

18.1 Overview ¶

Ollama Buddy can drive structured LLM output to generate simply-annotate annotation databases across whole directories of source code. The batch workflow attaches one file at a time, forces the model into a JSON schema via the “Annotate Project” skill, and applies the returned JSON to a .simply-annotations.el database at the project root.

18.2 Commands ¶

M-x ollama-buddy-annotate-directory

Or /annotate-dir. Prompts for a directory, queues every matching source file (see ollama-buddy-annotate-directory-extensions), and sends one annotation request per file. A post-response hook auto-applies the returned JSON and dequeues the next file. With a prefix argument, descends recursively.

M-x ollama-buddy-annotate-directory-cancel

Or /annotate-cancel. Stops an in-flight batch, removes the post-response hook, and clears the queue.

M-x ollama-buddy-annotate-apply-last-response

Or C-c A in the chat buffer, or /annotate. Applies the JSON from the last response in the chat buffer to the annotation database for a file you pick. Used both manually and internally by the batch loop.

18.3 Context-aware batching ¶

Each queued file’s estimated token cost (characters / 4) is checked against the current model’s context window. Files whose estimate exceeds the budget are skipped with a message. The reserve for the response is configurable:

ollama-buddy-annotate-context-reserve

Fraction of the context window reserved for the response and overhead (default 0.5). A file is allowed when estimated-tokens <= context-size * (1 - reserve).

18.4 Structured output schema ¶

The batch loads the “Annotate Project” user-prompt skill (configurable via ollama-buddy-annotate-directory-skill), which sets the active system prompt and installs a JSON schema via its #+FORMAT: header. The schema requires one annotations array whose items have sl, el, text, and tag (one of file, defun, or line). Structured output guarantees the model returns exactly this shape, which makes unattended batching reliable.

You can inspect the active schema at any time with C-c C-s (ollama-buddy-show-system-prompt-info) – the system-prompt buffer now shows the loaded #+FORMAT: schema alongside the prompt content when structured output is armed.

18.5 Database shape ¶

Annotations are written as full simply-annotate thread structures (id, created, status, priority, tags, comments) rather than plain strings, so the tag from the schema lands in the thread’s tags list directly. When the database strategy is project (default), file keys are stored relative to the project root to keep the database portable between clones.

ollama-buddy-annotate-db-strategy

Where to write the database. project (default) writes to .simply-annotations.el at the project root; global writes to simply-annotations.el in user-emacs-directory and stores absolute paths.

ollama-buddy-annotate-directory-extensions

List of file-name extensions considered by ollama-buddy-annotate-directory. Covers common compiled and scripting languages; extend it for project-specific file types.

18.6 Post-response hook ¶

The batch loop is built on ollama-buddy-post-response-hook, fired from the normal (non-multishot, non-tool) stream-completion path. Each hook function receives the model name as its argument. The hook is available to other automations that need to react when a response lands.

21.1 Overview ¶

Reasoning models (such as DeepSeek-R1, QwQ, and others) produce an internal chain-of-thought before giving their final answer. Ollama Buddy renders this reasoning content as a native org-mode heading that folds automatically, so the thinking process is available for inspection but does not clutter the conversation.

21.2 How It Works ¶

When a model emits <think> tags or uses the DeepSeek message.thinking API field, Ollama Buddy:

  1. Inserts a *** ✦ Think org heading in the chat buffer.
  2. Streams the reasoning content under that heading in real time.
  3. When reasoning ends, folds the heading with outline-hide-subtree and inserts a *** ✦ Response heading to mark the start of the actual answer.

The ✦ symbol appears in the header line and prompt line when thinking is enabled for the active model. When thinking is disabled (via /think), the symbol changes to ✧.

21.3 Key Bindings ¶

TAB

On a *** Think heading, unfold to peek at thinking content; press again to re-fold (standard org-mode behavior). Works during streaming and after completion.

21.4 Configuration ¶

ollama-buddy-thinking-enabled

When non-nil (default t), the Ollama API think parameter is sent to reasoning-capable models so they produce a thinking block. When nil, models skip reasoning entirely for faster, cheaper responses. Toggle with /think, transient T, or M-x ollama-buddy-toggle-thinking. The header line shows ✦ when enabled or ✧ when disabled.

ollama-buddy-collapse-thinking

When non-nil (default t), thinking content is automatically collapsed to the *** ✦ Think heading after the reasoning block ends.

ollama-buddy-hide-reasoning

When non-nil and ollama-buddy-collapse-thinking is nil, the thinking block is hidden entirely (replaced by “Thinking…”) rather than kept in the buffer.

ollama-buddy-thinking-models

Static list of model name substrings that are known reasoning models. Auto-detection from the Ollama /api/show capabilities array is also performed.

23 In-Buffer Replace ¶

  • Overview

  • Basic Workflow
  • Inline Diff View

  • Automatic Output Polishing
  • Implementation Notes

23.1 Overview ¶

In-buffer replace streams LLM output directly into the source buffer rather than the chat buffer. When active, commands that operate on a selected region (Refactor, Proofread, Git Commit, Dictionary Lookup, Synonym, Describe Code, or any custom role command) replace the selection in-place with the model’s response, shown with a green highlight pending your decision.

The existing chat buffer workflow is completely unchanged when the toggle is off (the default). All existing commands, prompts, system prompts, and model selection logic are unaffected — only the output destination changes.

Per-command override: individual commands can pin their destination regardless of the global toggle by setting the :destination property in ollama-buddy-command-definitions. A command with :destination in-buffer always rewrites in-buffer; one with :destination chat always opens the chat buffer. Commands without a :destination property continue to honour the global toggle. See Per-Command Response Destination, for details.

23.2 Enabling In-Buffer Replace ¶

Toggle via the transient Actions menu (C-c O, key W) or: 

M-x ollama-buddy-toggle-in-buffer-replace

The ✎ symbol appears in the header line and prompt line while the mode is active. The custom role menu footer shows a live ‘In-Buffer Replace [ON] / [OFF]’ toggle.

23.3 Basic Workflow ¶

  1. Toggle in-buffer replace on (C-c O ⇒ W).
  2. In any source buffer, select a region of text (e.g. a function body, a paragraph).
  3. Invoke a role command (e.g. Refactor from the Developer role).
  4. A dimmed ‘[Rewriting...]’ placeholder replaces the selection while connecting.
  5. The model’s response streams in with a green highlight.
  6. When streaming finishes, the mode line shows ‘ [Rewrite?]’.
  7. Press C-c C-c to accept (keep new text, remove highlight).
  8. Press C-c C-k to reject (discard new text and restore the original).

Cancelling mid-stream with C-c C-k stops the network process and restores the original selection immediately.

23.4 Inline Diff View ¶

After streaming completes, press C-c d to open an inline diff view inside the same buffer: 

[new text — green background]
  [changed words — brighter green overlay]

── Original ────────────────────────────
[original text — yellow background]
  [removed words — red strikethrough overlay]
────────────────────────────────────────

Word-level differences are highlighted by smerge-refine-regions: green overlays mark added or changed words in the new text; red strikethrough overlays mark words present in the original but absent from the rewrite.

Press C-c d again to hide the diff block without accepting or rejecting. Accepting (C-c C-c) or rejecting (C-c C-k) automatically removes the diff block.

23.5 Key Bindings (while rewrite is pending) ¶

C-c C-c

Accept the rewrite — keep the streamed text and remove the highlight.

C-c C-k

Reject the rewrite — stop streaming (if still in progress) and restore the original selected text.

C-c d

Toggle the inline diff view. Shows the original text below the new text with word-level smerge highlighting. Press again to hide.

23.6 Automatic Output Polishing ¶

Two automatic adjustments help keep in-buffer results clean:

  • Clean-output tone — when in-buffer replace is active a “Return only the requested content” instruction is automatically prepended to the effective system prompt, suppressing introductory phrases and commentary.
  • Code-fence stripping — if the model wraps its response in markdown code fences (‘```lang … ```’), the fences are stripped before insertion, leaving only the inner content.

23.7 Implementation Notes ¶

The feature is implemented in ollama-buddy-rewrite.el, loaded optionally: 

(require 'ollama-buddy-rewrite nil t)

The module creates a direct make-network-process connection to Ollama (/api/chat) and streams JSON responses using the same line-delimited format as the main chat buffer filter. Two Emacs markers track the rewrite region:

  • A non-advancing start marker (type nil) — stays before inserted text.
  • An advancing end marker (type t during streaming, switched to nil on completion) — moves past each newly inserted chunk, then fixed so the diff block insertion does not displace it.

The inline diff block is inserted as real buffer text at the fixed end marker position. smerge-refine-regions is called with both the original block and the new text residing in the same buffer, which is the standard within-buffer usage of that function. Original text is preserved in the rewrite state plist for rejection; no undo history manipulation is required.

23.8 Limitations ¶

  • Only works with local Ollama models (not remote providers).
  • Only one rewrite can be pending at a time.
  • The streaming response is inserted verbatim; no markdown-to-org conversion is performed (unlike the chat buffer) — the fence stripper handles the common code-block case.

24.1 Overview ¶

Tool calling (also known as function calling) enables LLMs to invoke Emacs functions during conversations. Instead of just generating text, a tool-capable model can read files, list directories, search buffers, perform calculations, and execute shell commands — all orchestrated automatically within the conversation flow.

The tool calling framework is provided by ollama-buddy-tools.el.

24.2 Setup ¶

;; Load the tools module
(require 'ollama-buddy-tools)

;; Enable tool calling
(setq ollama-buddy-tools-enabled t)

You must use a model that supports tool calling. Tool-capable models are indicated with a ⚒ symbol in the header line, model selection, and model management buffer.

24.3 Built-in Tools ¶

  • Safe Tools (available in safe mode)
  • Unsafe Tools (require safe mode off)

24.3.1 Safe Tools (available in safe mode) ¶

read_file

Read the contents of a file from the filesystem.

list_directory

List the contents of a directory.

get_buffer_content

Get the content of an Emacs buffer by name.

list_buffers

List all open Emacs buffers, with optional regex filtering.

calculate

Evaluate a mathematical expression using Emacs calc-eval.

search_buffer

Search for a regex pattern in a buffer and return matching lines.

search_files

Grep for a regex pattern across files in a directory tree. Supports an optional file name glob filter (e.g. "*.el") and a configurable result limit. Skips binary files and hidden directories.

get_region

Get the currently selected text (active region) along with the buffer name and line range.

get_buffer_info

Get metadata about an Emacs buffer (file path, major mode, point position, line count, modified and read-only state, narrowing) without returning the full buffer contents.

24.4 Example Prompts ¶

With a tool-capable model selected and tools enabled, try prompts like: 

What files are in my ~/Documents directory?

Read the file ~/.emacs.d/init.el and summarize what packages I'm using.

Search for all TODO comments in my project under ~/src/myapp/

What is 2^64 - 1?

Explain the selected code.

What does the function `mapcar' do in Emacs Lisp?

For unsafe tools (with safe mode disabled): 

Run df -h and tell me how much disk space I have.

Create a file at /tmp/hello.txt with Hello World.

Evaluate (length (buffer-list)) and tell me how many buffers I have.

24.5 How It Works ¶

When tool calling is enabled and a tool-capable model is selected, the conversation flow works as follows:

  1. You send a prompt (e.g., “What files are in my home directory?”).
  2. Ollama Buddy sends the prompt along with the tool schemas to the model.
  3. The model decides whether to call a tool. If so, it returns a tool call request instead of text.
  4. Ollama Buddy executes the requested tool (e.g., list_directory with path ~).
  5. The tool result is sent back to the model as a tool role message.
  6. The model generates a natural language response incorporating the tool output.
  7. Steps 3–6 may repeat if the model needs multiple tools, up to ollama-buddy-tools-max-iterations.

This loop is transparent to the user — you see the final response in the chat buffer.

  • Model Compatibility

24.5.1 Model Compatibility ¶

Tools are only sent to models that support them. If you have tools enabled but switch to a model that doesn’t support tool calling:

  • The request proceeds normally without tools (no error)
  • The header line ⚒ indicator disappears (tools enabled but not applicable)
  • The prompt line ⚒ indicator also disappears (model doesn’t support tools)
  • When you switch back to a tool-capable model, tools become active again

This allows you to leave tools enabled globally while freely switching between models.

24.6 Configuration ¶

ollama-buddy-tools-enabled

Whether tool calling is active (default: nil).

ollama-buddy-tools-safe-mode

When non-nil, only safe (read-only) tools can execute (default: t).

ollama-buddy-tools-auto-execute

When non-nil, tools execute without confirmation prompts (default: t). When nil, each tool invocation shows a yes-or-no-p prompt before executing.

ollama-buddy-tools-unguarded

When non-nil, bypass ALL safety prompts and let tools run to completion (default: nil). This disables confirmation for unsafe tools (e.g. execute_shell), large result warnings, and safe-mode restrictions. Auto-execute is implied. The header line turns red and shows ☠ as a visual warning.

ollama-buddy-tools-max-iterations

Maximum tool-call round-trips per user prompt to prevent infinite loops (default: 10).

ollama-buddy-tools-builtin-enabled

Whether to load the 8 built-in tools on startup (default: t). When nil, only custom user-registered tools are available.

ollama-buddy-tools-models

List of model names known to support tool calling. Used for the ⚒ indicator. 

(setq ollama-buddy-tools-models
  '("qwen3" "qwen3-coder" "llama3.1" "llama3.3"
    "mistral" "gpt-oss" "command-r"))
ollama-buddy-tools-large-result-threshold

Token count threshold for warning about large tool results (default: 2000). When a tool returns a result exceeding this threshold, the user is prompted with options to (p)roceed, (t)runcate to the threshold, or (c)ancel — even when auto-execute is enabled. Set to 0 to disable the guard.

24.7 Registering Custom Tools ¶

Use ollama-buddy-tools-register to add your own tools: 

(ollama-buddy-tools-register
 'my-tool                          ; NAME (symbol or string)
 "Description of what the tool does" ; DESCRIPTION
 '((type . "object")               ; PARAMETERS (JSON schema)
   (required . ["param1"])
   (properties . ((param1 . ((type . "string")
                              (description . "Parameter description"))))))
 (lambda (args)                    ; FUNCTION (receives alist, returns string)
   (let ((param1 (alist-get 'param1 args)))
     (format "Result: %s" param1)))
 t)  ; SAFE flag: t = safe (read-only), nil = unsafe

To remove a tool: 

(ollama-buddy-tools-unregister 'my-tool)

The tool function receives an alist of arguments parsed from the model’s JSON response and must return a string result. If the function returns a non-string value, it is formatted with %S.

25.1 Overview ¶

Plan mode provides a structured task planning workflow using org-mode TODO headings. Instead of asking the LLM to do everything at once, you describe a task and the LLM generates a multi-step plan. You can then review, edit, skip, or execute steps individually — with full tool calling support.

The plan module is provided by ollama-buddy-plan.el and is loaded automatically.

25.2 Workflow ¶

  1. Type /plan in the prompt area to activate plan mode.
  2. Describe your task and send (C-c C-c). The LLM generates a plan: 
    ** TODO Step 1: Create user model
Define the schema with email, password hash, and timestamps.

** TODO Step 2: Add registration endpoint
Implement POST /register with input validation.

** Notes
Consider rate-limiting login attempts.
  3. Review the plan. Use S-<right>/S-<left> or C-c C-t on any step heading to cycle its TODO state.
  4. Execute steps with /plan-next (one at a time) or /plan-all (all remaining). The current step is marked DOING and a progress indicator is inserted in the buffer.
  5. After a step completes, use /plan-done to mark it DONE.
  6. When finished, use /plan-stop to deactivate plan mode and restore your previous system prompt.

25.3 TODO States ¶

Plan mode configures a custom org TODO sequence:

TODO

Step has not started (red).

DOING

Step is currently being executed (orange).

DONE

Step completed successfully (green). This is a “done” state in org terms.

SKIP

Step was skipped by the user (grey). This is also a “done” state.

States cycle with S-<right>: TODO → DOING → DONE → SKIP.

25.4 Slash Commands ¶

/plan

Activate plan mode. Sets the planning system prompt and configures TODO states.

/plan-next

Execute the next TODO step. Marks it DOING, inserts a progress indicator, switches to a minimal execution prompt, and sends the step to the LLM. If tools are enabled, the LLM can use them to carry out the work.

/plan-all

Execute all remaining TODO steps in sequence.

/plan-done

Mark the current DOING step as DONE and restore the planning prompt.

/plan-status

Show a progress summary in the echo area (e.g. “Plan: 2/6 done, 1 doing, 3 remaining, 0 skipped”).

/plan-stop

Deactivate plan mode. Restores the previous system prompt and org TODO keywords.

25.5 Tool Calling Integration ¶

When tools are enabled (/tools), plan step execution uses the LLM’s native tool calling mechanism. During execution, the system prompt is temporarily switched to a minimal one-line prompt to avoid interfering with tool invocation. The planning prompt is restored when you mark the step done with /plan-done.

25.6 Customization ¶

ollama-buddy-plan-todo-states

List of TODO states for plan steps. Default: ("TODO" "DOING" "DONE" "SKIP").

ollama-buddy-plan-auto-advance

When non-nil (default t), report progress after marking a step done.

26.1 Overview ¶

RAG allows you to index local documents and source code, then retrieve relevant context for conversations via semantic search. Instead of pasting entire files into prompts, RAG automatically finds the most relevant chunks and attaches them as context, keeping token usage efficient and responses grounded in your actual documents.

The RAG module (ollama-buddy-rag.el) is loaded by default.

26.2 Requirements ¶

By default, RAG uses Ollama for embeddings and requires an embedding model to be pulled: 

ollama pull nomic-embed-text

Other Ollama embedding models include mxbai-embed-large and all-minilm. You can pull these from Model Management (C-c M) under the “Embedding (RAG)” category.

Alternatively, you can point RAG at any OpenAI-compatible embedding server instance) by setting ollama-buddy-rag-embedding-base-url and ollama-buddy-rag-embedding-api-style. In that case no Ollama embedding model is needed.

26.3 Quick Start ¶

  1. Pull an embedding model: ollama pull nomic-embed-text
  2. Open the RAG menu: C-c O then r
  3. Index a directory: select i and choose a directory
  4. Wait for indexing to complete (progress shown in status line); press c to pause if needed, r to resume later
  5. Search your index: select s and enter a query
  6. Or attach context to a conversation: select a, enter a query, then chat normally
  7. After editing files, update the index incrementally: select u and choose the same directory

26.4 How It Works ¶

  1. Indexing: Files in a directory are read, split into chunks (by paragraph/section), and each chunk is sent to the embedding model to produce a vector representation. The resulting index is saved to disk.
  2. Searching: Your query is embedded using the same model, then compared against all chunk vectors using cosine similarity. The most relevant chunks are returned.
  3. Attaching: Retrieved chunks are injected into the conversation context as a system-level prefix, so the LLM can reference them when generating a response.

26.5 Configuration ¶

ollama-buddy-rag-embedding-model

The embedding model to use (default: "nomic-embed-text").

ollama-buddy-rag-chunk-size

Target number of tokens per chunk (default: 300).

ollama-buddy-rag-chunk-overlap

Number of overlapping tokens between consecutive chunks (default: 50).

ollama-buddy-rag-top-k

Number of top results to return from a search (default: 5).

ollama-buddy-rag-similarity-threshold

Minimum cosine similarity score for a chunk to be included in results (default: 0.3).

ollama-buddy-rag-index-directory

Directory where RAG index files are stored (default: ~/.emacs.d/ollama-buddy-rag/).

ollama-buddy-rag-file-extensions

List of file extensions to include when indexing a directory (default: common text and code extensions).

ollama-buddy-rag-exclude-patterns

List of glob patterns to exclude from indexing (default: ("node_modules" ".git" "__pycache__" ".elc")).

ollama-buddy-rag-batch-size

Number of chunks to embed per API call (default: 10).

ollama-buddy-rag-checkpoint-interval

Save partial progress every N batches during indexing (default: 5). Set to 1 to save after every batch, or 0 to disable automatic checkpointing (progress is still saved on pause).

ollama-buddy-rag-embedding-base-url

Base URL for the embedding service. When nil (the default), the main Ollama host and port are used. Set this to point at a separate OpenAI-compatible embedding server, for example: 

(setq ollama-buddy-rag-embedding-base-url "http://localhost:9000")

This is useful when running a dedicated embedding service independently from your Ollama instance.

ollama-buddy-rag-embedding-api-style

The API style used by the embedding endpoint. Two values are supported:

ollama

(default) POST to /api/embed — the Ollama native endpoint.

openai

POST to /v1/embeddings — compatible with any OpenAI-style embedding service. Use this together with ollama-buddy-rag-embedding-base-url.

ollama-buddy-rag-embedding-api-key

Optional Bearer token sent to the embedding service. Only needed when ollama-buddy-rag-embedding-api-style is openai and the service requires authentication. Leave nil for local servers.

26.6 PDF Support ¶

RAG can index PDF files if pdftotext (from poppler-utils) is installed on your system. PDF support is auto-detected at runtime:

  • If pdftotext is available, .pdf files are included when indexing directories.
  • If pdftotext is not installed, PDF files are silently skipped.
  • PDFs bypass the file size limit (ollama-buddy-rag-max-file-size) since binary PDF size does not reflect extracted text size.

To install pdftotext: 

# Debian/Ubuntu
sudo apt install poppler-utils

# Arch Linux
sudo pacman -S poppler

# macOS
brew install poppler

26.7 Interactive Commands ¶

These commands are available from the RAG transient menu (C-c O then r):

ollama-buddy-rag-index-or-update-directory (i)

Index or update a directory. If an index already exists for the directory, it is incrementally updated: only files that have been added or modified since the last indexing are re-chunked and re-embedded, chunks from unchanged files are retained, and chunks for deleted files are removed. If no existing index is found, a full index is created. If the embedding model has changed since the index was created, a full re-index is required and the user is prompted before proceeding.

ollama-buddy-rag-pause (c)

Pause an in-progress indexing or update operation and save partial progress to disk. The pause takes effect before the next embedding batch is sent, so the current in-flight batch completes but no further batches are processed. The partial index is saved with all already-embedded chunks and can be resumed later. ollama-buddy-rag-cancel is an alias for this command.

ollama-buddy-rag-resume (r)

Resume a previously paused indexing operation. Loads the partial index, retains all already-embedded chunks, and continues embedding the remaining pending chunks. If only one partial index exists it is selected automatically; otherwise you are prompted to choose. The embedding model must match the one used when the index was created.

ollama-buddy-rag-search (s)

Search the index with a text query. Results are displayed in an org-mode buffer showing similarity scores, file paths, and content excerpts.

ollama-buddy-rag-attach (a)

Search and attach the top results as context for the next conversation. Attached RAG context appears when you use Show Attachments.

ollama-buddy-rag-list-indexes (w)

List all saved RAG indexes with their chunk counts, file counts, and whether the index supports incremental updates.

ollama-buddy-rag-delete-index (d)

Delete a saved RAG index.

26.8 Incremental Updates ¶

When you first index a directory with i, the index records modification times for every file. On subsequent runs you can use u (Update Directory) instead of re-indexing from scratch:

  1. Files whose modification time has changed are re-chunked and re-embedded.
  2. New files are chunked and embedded.
  3. Chunks belonging to deleted files are removed.
  4. Chunks for unchanged files are kept as-is — no embedding calls are made for them.

This makes updates significantly faster for large directories where only a few files have changed.

Incremental updates require the same embedding model that was used to create the index. If you change ollama-buddy-rag-embedding-model, a full re-index is needed (you will be prompted).

Indexes created before version 3.6.0 do not contain file metadata. Running Index/Update Directory on such an index automatically falls back to a full re-index and saves the metadata for future incremental updates.

26.9 Pause and Resume ¶

Indexing large directories can take a long time. If you need to close Emacs or stop the process, you can pause and resume later.

Pause

Press c in the RAG transient menu (or run M-x ollama-buddy-rag-pause). The current batch completes, then all progress is saved to disk as a partial index. The partial index is immediately searchable for any chunks that have already been embedded.

Resume

Press r in the RAG transient menu (or run M-x ollama-buddy-rag-resume). The partial index is loaded and embedding continues from where it stopped. When all remaining chunks are processed, the index is saved as a complete index.

Automatic checkpointing

Progress is also saved periodically during indexing (every 5 batches by default, controlled by ollama-buddy-rag-checkpoint-interval). This means even an unexpected Emacs crash preserves most of the work done so far. The next ollama-buddy-rag-resume picks up from the last checkpoint.

Partial index visibility

M-x ollama-buddy-rag-list-indexes marks partial indexes with [partial: N/M chunks] in the heading and shows a count of pending chunks.

Partial index handling

Running M-x ollama-buddy-rag-index-or-update-directory on a directory with a partial index will offer to resume it or start a fresh full re-index.

26.10 Inline @rag() Syntax ¶

You can use @rag(query) directly in prompts to automatically search your RAG index and attach results as context — no separate attach step needed. This mirrors the @search(query) syntax for web search. 

Tell me about @rag(giant squid attack) from the book.

Multiple queries are supported: 

Compare @rag(captain nemo) with @rag(ned land) in terms of character development.

How it works:

  1. The @rag() patterns are extracted from the prompt.
  2. Each query is embedded synchronously and searched against the index.
  3. Matching chunks are attached as RAG context.
  4. The @rag() wrappers are removed, preserving the query text in the prompt sent to the model.

If you have multiple indexes, you will be prompted to select one. If only one index exists, it is used automatically.

26.11 Example Workflow ¶

Suppose you have the text of “Twenty Thousand Leagues Under the Sea” in ~/books/leagues.txt: 

;; 1. Index the book
C-c O r i  ->  ~/books/

;; 2a. Attach via transient menu
C-c O r a  ->  "giant squid attack"

;; 2b. Or use inline syntax -- no separate step needed
"Describe the @rag(giant squid encounter) in detail, referencing the text."

;; 3. Ask a question in chat -- the LLM now has relevant passages
"Describe the giant squid encounter in detail, referencing the text."

Without RAG, the model would give a generic answer from training data. With RAG, it references specific passages from your indexed document.

27.1 Managing Token Usage ¶

:TIMESTAMP: [2026-03-16 Sun 14:32]
:TOKENS:   142
:RATE:     23.4
:ELAPSED:  6.1s
:WAIT:     2.1s
:END:

You can use org-columns to get a tabular overview of all responses in a session. The data survives session save/load since it is in the buffer text.

  • mistral:latest - Good balance of performance and quality

  • Medium models (13B) - 16GB minimum, 24GB+ recommended
  • Large models (34B+) - 32GB+ recommended

30.1.1 Connection Problems ¶

Symptom: Unable to connect to Ollama server
  • Check if Ollama is running with ps aux | grep ollama
  • Verify host and port settings (ollama-buddy-host and ollama-buddy-port)
  • Try connecting to Ollama directly: curl http://localhost:11434/api/tags
Symptom: Connection breaks during long responses
  • This can happen with very large responses
  • Try setting a lower num_predict value
  • Check if your OS has any network timeout settings

30.1.2 Model Problems ¶

Symptom: Model loads but gives poor responses
  • Try a different model
  • Adjust parameters (increase temperature for more creativity)
  • Provide clearer or more detailed prompts
  • Check if the model is appropriate for your task
Symptom: Model fails to load or crashes
  • Check system memory usage
  • Try a smaller quantized model
  • Adjust num_ctx to a smaller value
  • Set low_vram to t if using GPU

30.1.3 Interface Issues ¶

Symptom: Chat buffer becomes unresponsive
  • Cancel any running requests with C-c k
  • Check if Emacs is using high CPU
  • Try disabling token statistics display
  • Close and reopen the chat buffer
Symptom: Markdown conversion issues
  • Toggle markdown conversion off with C-c C-o
  • Check if the response contains complex formatting
  • Try editing the history to fix formatting issues

30.2 Debugging ¶

  • Enable Debug Mode
  • Check Logs
  • Report Issues

30.2.1 Enable Debug Mode ¶

To get more information about what’s happening: 

M-x ollama-buddy-toggle-debug-mode

This opens a debug buffer showing raw JSON communication with Ollama.

30.2.2 Check Logs ¶

Ollama logs can be useful for troubleshooting: 

tail -f ~/.ollama/logs/ollama.log

30.2.3 Report Issues ¶

If you encounter a bug:

  1. Enable debug mode
  2. Reproduce the issue
  3. Copy the debug output
  4. Report the issue on GitHub with:
    • Emacs version
    • Ollama version
    • Model used
    • Debug output
    • Steps to reproduce

31.5.1 User System Prompts Issues ¶

Symptom: Cannot save system prompts
  • Check if ‘ollama-buddy-user-prompts-directory‘ exists and is writable
  • Ensure you have set a system prompt before trying to save it
  • Check for any error messages in the minibuffer or *Messages* buffer
Symptom: Prompts not appearing in the load menu
  • Verify that prompts are saved in the correct format (category__title__system.org)
  • Check if the prompts directory contains files with proper formatting
  • Try refreshing the prompts cache with ‘M-x ollama-buddy-user-prompts–refresh-cache‘
