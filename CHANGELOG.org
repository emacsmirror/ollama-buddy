** <2026-04-14 Tue> *7.4.1*

Response post-processing hook and a thinking-block point-anchor fix.

*** Features

- *=ollama-buddy-response-post-process-functions=* -- New hook run on the streamed response region just before markdown-to-org conversion.  Each function is called with =(START END)= buffer positions and may edit the region freely.  Useful for normalising output from models that concatenate structured content into a single line.
- *=ollama-buddy--normalize-numbered-lists=* -- Default entry on the new hook.  Splits glued numbered-list items of the form =N. [CATEGORY] ...= onto their own lines when the model emits them as one paragraph (notably cloud gemma).  Regex is deliberately narrow -- it only fires when a bracketed uppercase tag follows the number, so prose like "see rule 1.  Then rule 2." is left alone.  Idempotent on already-clean input.
- *Emacs Blog Reviewer prompt* -- New =writing__Emacs-Blog-Reviewer__system.org= user prompt.  Reviews an Emacs blog post for grammar and technical accuracy (function and variable name spelling, key-binding notation, elisp syntax, deprecated APIs, version claims, core vs third-party assertions).  Output format is a numbered one-line-per-issue list so dense cloud models cannot collapse fields into a paragraph.

*** Bug Fixes

- *Thinking-block point anchor* -- Fixed the first response token being inserted at the top of the folded thinking block instead of after =*** Response=.  =ollama-buddy--finalize-thinking-block= deletes the raw streamed thinking region and reinserts clean content plus the response heading, which collapses the outer =save-excursion= marker into the new content and leaves =(point)= inside the fold.  Both the API-style (=message.thinking=) and tag-style (=</think>=) paths now re-anchor to =(point-max)= immediately after finalize returns.

** <2026-04-13 Mon> *7.4.0*

Directory-scale project annotation via structured output.

*** Features

- *=/annotate-dir=* -- New slash command and =ollama-buddy-annotate-directory= iterate every source file under a directory, send one annotation request per file, and auto-apply the returned JSON to =.simply-annotations.el= without manual intervention.  Prefix arg descends recursively.  Cancel an in-flight batch with =/annotate-cancel= or =ollama-buddy-annotate-directory-cancel=.

- *Post-response hook* -- New =ollama-buddy-post-response-hook=, fired from the normal (non-multishot, non-tool) stream-completion path.  Used by =annotate-directory= to chain successive requests and available for other automations that need to react when a response lands.
- *=C-c A=* -- New chat-buffer binding for =ollama-buddy-annotate-apply-last-response=, so applying a one-off annotation response no longer requires typing the slash command.
- *New cloud models* -- Added =deepseek-v3.2:cloud=, =gemini-3-flash-preview:cloud=, =qwen3-next:80b-cloud=, and =nemotron-3-super:cloud= to =ollama-buddy-cloud-models=.

** <2026-04-09 Thu> *7.1.3*

Fix process-filter crash on cloud usage limit responses.

*** Bug Fixes

- *Cloud limit crash* -- When an Ollama cloud model session was at 100% usage, the non-2xx error response from =/api/chat= could begin with a chunked-transfer-encoding chunk-size prefix (e.g. =82\r\n{...}=).  =json-read-from-string= happily parsed the leading =82= as the integer =82=, which then crashed =ollama-buddy--handle-http-error= with =Wrong type argument: listp, 82= inside the network-process filter, repeating for every subsequent chunk.  The streaming filter now strips any leading non-={= bytes before parsing and requires the parsed value to be a cons before treating it as an alist.  =handle-http-error= itself is also defensive against non-alist input, falling back to a generic =HTTP NNN= message (or the raw string body) so a future caller can't reintroduce the crash.  The same =consp= guard is mirrored in the curl backend filter for symmetry.

** <2026-03-14 Fri> *4.0.0*

Generic provider registration system for external LLMs.

*** Breaking Changes

- *External LLM providers now use a generic registration system* -- A new =ollama-buddy-provider.el= module provides =ollama-buddy-provider-create=, a single factory function for registering any external LLM provider.  Three API types are supported: =openai= (default), =claude=, and =gemini=.

  *Migration:* The existing individual provider files (=ollama-buddy-openai.el=, =ollama-buddy-claude.el=, etc.) have been rewritten as thin wrappers around the new system.  *Your current configuration with =(require 'ollama-buddy-openai)= and =(setq ollama-buddy-openai-api-key ...)= will continue to work unchanged.* However, these wrapper files are deprecated and will be removed in a future release.

  *New approach (recommended):*

  #+begin_src elisp
  (require 'ollama-buddy-provider)

  (ollama-buddy-provider-create
   :name "OpenAI" :prefix "a:"
   :api-key (lambda () (auth-source-pick-first-password
                        :host "ollama-buddy-openai" :user "apikey"))
   :endpoint "https://api.openai.com/v1/chat/completions"
   :models-endpoint "https://api.openai.com/v1/models"
   :models-filter (lambda (id) (string-match-p "gpt" id)))

  (ollama-buddy-provider-create
   :name "Claude" :prefix "c:" :api-type 'claude
   :api-key (lambda () (auth-source-pick-first-password
                        :host "ollama-buddy-claude" :user "apikey"))
   :endpoint "https://api.anthropic.com/v1/messages"
   :models-endpoint "https://api.anthropic.com/v1/models")

  (ollama-buddy-provider-create
   :name "Gemini" :prefix "g:" :api-type 'gemini
   :api-key (lambda () (auth-source-pick-first-password
                        :host "ollama-buddy-gemini" :user "apikey"))
   :endpoint "https://generativelanguage.googleapis.com/v1beta/models/%s:generateContent"
   :models-endpoint "https://generativelanguage.googleapis.com/v1/models"
   :models-filter (lambda (id) (string-match-p "gemini" id)))
  #+end_src

  Any OpenAI-compatible server (LM Studio, llama.cpp, vLLM) can be registered
  with a single call -- no new file required.

*** New Features

- *=ollama-buddy-provider-create=* -- Register any LLM provider with one function call.  Supports static model lists, dynamic model discovery with optional filters, and lazy API key resolution via function values.
- *=ollama-buddy-provider-remove=* / *=ollama-buddy-provider-refresh=* / *=ollama-buddy-provider-list=* -- Manage registered providers at runtime.
- *Welcome screen shows generic providers* -- Providers registered via =ollama-buddy-provider-create= now appear in the welcome screen alongside those loaded via the legacy =require= approach.

*** Deprecated

- *Individual provider files* -- =ollama-buddy-openai.el=, =ollama-buddy-claude.el=, =ollama-buddy-gemini.el=, =ollama-buddy-grok.el=, =ollama-buddy-codestral.el=, =ollama-buddy-openrouter.el=, and =ollama-buddy-deepseek.el= are now thin wrappers.  They still work but will be removed in a future release.  =ollama-buddy-copilot.el= and =ollama-buddy-openai-compat.el= are unaffected.

*** New Features

*** New Features

** <2026-02-05 Thu> *1.3*

Added Web Search Integration

This release introduces real-time web search capabilities through Ollama's Web Search API, allowing LLMs to access current information from the web.

*** Web Search Pipeline

The web search feature implements a multi-stage pipeline:

1. *Query to Ollama API*: Search queries are sent to =https://ollama.com/api/web_search= via REST API with Bearer token authentication
2. *URL Extraction*: The API returns search results with URLs (the content snippets from the API are currently ignored in favor of fetching full page content)
3. *eww/shr Processing*: Each URL is fetched and processed through Emacs' built-in eww/shr HTML renderer, which:
   - Parses the HTML using =libxml-parse-html-region=
   - Extracts the main content (looking for =<article>=, =<main>=, or content divs)
   - Renders to clean plain text using =shr-insert-document=
   - Escapes org-mode special characters (=*= and =#= at line starts become =,*= and =,#=)
4. *Context Attachment*: The cleaned text is formatted with org headings and attached to the conversation context
5. *LLM Submission*: The search results are included when sending prompts to any configured LLM provider

*** Features

- Added =ollama-buddy-web-search.el= module
- Inline search syntax: =@search(query)= in prompts automatically triggers search
- Manual commands:
  - =M-x ollama-buddy-web-search= - Search and display results in a buffer
  - =M-x ollama-buddy-web-search-attach= - Search and attach to conversation context
- Keybindings: =C-c / s= (search), =C-c / a= (attach)
- Transient menu: =C-c O /= for web search operations
- Async URL fetching for better performance
- Status line shows ♁N indicator when web searches are attached
- Web searches integrated into attachment system:
  - =C-c C-w= shows both file attachments and web searches
  - =C-c 0= clears all attachments including web searches
- Full content display in attachment viewer with org-mode folding

*** Provider Support

Web search works with all LLM providers:
- Local Ollama models (=o:= prefix)
- OpenAI (=a:= prefix)
- Anthropic Claude (=c:= prefix)
- Google Gemini (=g:= prefix)
- X Grok (=k:= prefix)
- GitHub Copilot (=p:= prefix)
- Mistral Codestral (=s:= prefix)

*** Configuration

#+begin_src elisp
;; Required: Get API key from https://ollama.com/settings/keys
(setq ollama-buddy-web-search-api-key "your-api-key")

;; Optional customizations
(setq ollama-buddy-web-search-max-results 5)        ; Number of URLs to fetch
(setq ollama-buddy-web-search-snippet-length 2000)  ; Max chars per result
(setq ollama-buddy-web-search-include-urls nil)     ; Include URLs in context
(setq ollama-buddy-web-search-content-source 'eww)  ; Content source (see below)

;; Load the module
(require 'ollama-buddy-web-search nil t)
#+end_src

*** Content Source Options

The =ollama-buddy-web-search-content-source= variable controls how content is retrieved:

- =eww= (default, recommended) - Fetches each URL and renders HTML through Emacs' eww/shr. Produces cleaner, more complete content but requires additional HTTP requests.

- =api= (experimental) - Uses content returned directly from the Ollama API without fetching individual URLs. Faster but content quality depends on what the API returns.

*** Example Usage

;; Multiple searches

#+end_src

(use-package ollama-buddy

   (auth-source-pick-first-password :host "ollama-buddy-gemini" :user "apikey"))

  (require 'ollama-buddy-openai nil t)
  (require 'ollama-buddy-claude nil t)
